# Fjord — Claude Code Context

Fjord is a Jellyfin media frontend built in Rust with Slint as the GUI toolkit and libmpv for video playback. It is built by KalasKonrad as a personal project, partly as a learning exercise in Rust and partly to solve a real problem: every existing Flutter-based Jellyfin frontend (Fladder, Jellyflix) uses media_kit which embeds mpv into a Flutter texture. That path never calls `mpv_render_context_report_swap()`, so mpv has no vsync feedback and playback is choppy on NVIDIA legacy drivers on Wayland. Fjord fixes this by using the mpv render API so mpv renders into an OpenGL FBO that Slint composites, with `report_swap()` called after every frame.

### `fjord-app/src/` module responsibilities

Each module owns one concern. `main.rs` wires all modules together: rendering
notifier, mpv event-poll timer, not-watched refresh timer, applying saved config
on startup, and `AppState::get(&window).on_*()` callback registrations.
`slint::include_modules!()` must stay in `main.rs` — it generates `MainWindow`
and `AppState` (the Slint global) as `crate::MainWindow` and `crate::AppState`.
Every module that accesses the global imports `use slint::Global;` and uses
`AppState::get(&window).set_X()` / `.get_X()` / `.on_X()`.

`show_toast(ww: Weak<MainWindow>, msg: String)` in `main.rs` is the canonical way to surface errors to the user. It is safe to call from any thread or from the Slint event loop. It sets `toast-message` + `toast-visible` via `invoke_from_event_loop`; a Slint `Timer` in `main.slint` auto-dismisses after 4 s.

## Key design decisions

### mpv render API (not X11 embedding)
mpv uses `vo=libmpv` and `mpv_render_context`. It never opens its own window. Each frame:
1. Slint's `BeforeRendering` notifier fires on the GL thread
2. mpv renders into the back FBO (`mpv_render_context_render`)
3. The FBO texture is exposed to Slint as a `BorrowedOpenGLTexture`
4. Slint's `AfterRendering` notifier calls `report_swap()` for vsync feedback

**Double-buffer FBO:** Two GL texture/FBO pairs alternate each frame. Single-buffer caused Slint to skip re-renders because the texture ID was unchanged (Slint's change detection). Alternating IDs force a re-render every frame.

**Drop ordering:** `MpvRenderCtx` must be dropped before `Player`. This is enforced in `VideoState` and in the rendering teardown path.

### Four playback modes
1. **Fullscreen player** (`is-playing = true`): covers the full window, shows controls bar + inline stats overlay.
2. **Video behind UI** (`has-background-player + video-behind-ui = true`): video fills the full window (dim overlay `#00000044`); all overlay screens (Detail/Series/Season/Person) have transparent root backgrounds and transparent inner header Rectangles so the video shows through everywhere; the 56px `MiniPlayerBar` is docked at the **bottom** above the music bar (if any).
3. **Mini-player bar** (`has-background-player && !is-playing`): full-width `MiniPlayerBar` docked at the **bottom** of the window. Mode 3 (!video-behind-ui): 108px — live video thumbnail (192×108 px) + title + Resume/Stop buttons. Mode 2 (video-behind-ui): 56px compact bar — NOW PLAYING label + title + FjordButton Resume/Stop. `bar-h` is 108px (mode 3) or 56px (mode 2) or 0px when player is off. `float-card-focused` (-1=none, 0=Resume, 1=Stop); `focus_bar_on_up` still used for keyboard access. Down/Back unfocuses.
4. **Music bar** (`is-audio-playing = true`): full-width `MusicPlayerBar` (72 px) docked at the very bottom. Shown when audio-only items play (`item_type == "Audio"`). `is-playing` is **not** set — no fullscreen player opens. Left zone: 60×60 album art + title + artist (click or Enter on focus slot 0 → `open-now-playing`, opens the fullscreen Now Playing screen — same target for mouse and keyboard). Centre: ⏸/▶ (slot 1) + ⏹ (slot 2) + progress bar (slot 3) + elapsed/total. Right zone: ⏮ Prev (slot 4) + ⏭ Next (slot 5) + ⇌ Shuffle (slot 6) + ↺ Repeat (slot 7) + ⋮ Queue (slot 8) + ♪ Lyrics (slot 9, only rendered when `lyrics-available`). Shuffle button accent when `queue-shuffle`; Repeat button accent when `queue-repeat-mode > 0`, shows "↺¹" for One. Lyrics button accent when `show-lyrics`. `music-bar-play-pause` delegates to `pause_play_toggle`; `music-bar-stop` calls `do_stop_playback`. Keyboard: `[`/`]` → PrevTrack/NextTrack (global, any mode when `is_audio_playing`); `q`/`Q` → OpenQueuePanel (opens QueuePanel overlay; also works while idle whenever `queue-count > 0`); `L`/`l` → ToggleLyrics (global when `is_audio_playing`; the standalone `LyricsView` overlay is suppressed while Now Playing is open since that screen already embeds its own synced lyrics panel — `!show-now-playing` added to its condition in `main.slint`, CR11-8); `m`/`M` → ToggleNowPlaying (global when `is_audio_playing`, any mode except ContextMenu/Player); focus slots navigate with arrow keys (Down→timeline, Up→unfocus from non-timeline); slot 8 (⋮) Right → 9 (♪) when `lyrics-available` else straight to 10 (🔉); slot 9 Right → 10; slots 10/11 (🔉/🔊, always shown) call the same `volume-up`/`volume-down` callbacks as the fullscreen player's Up/Down — Confirm nudges ±5 each press like real remote volume buttons; slot 11 Right absorbed.

5. **Now Playing** (`show-now-playing && is-audio-playing`, `NowPlayingScreen`): fullscreen music view covering the music bar. Layout: top bar (← Back + "Now Playing" title, same pattern as album/collection/artist screens); big 380px album art + title/artist + click-to-seek progress bar + transport row (left half); synced lyrics, reusing `lyrics-lines`/`lyrics-active-idx` with the same kb-y auto-scroll idiom and text sizing as `LyricsView` (40px active / 24px other, 76px MINIMUM line height, word-wrapped instead of elided so a long line is never cut off with "…"; each row's height is floored at what an invisible `sizer` probe — same text, always measured at the active 40px/700 styling — reports as `preferred-height`, so a line that will need 2 lines once highlighted already reserves that space while still small/inactive, and activating it only animates font-size/color rather than also reflowing the list — both surfaces share these values) (right half); "Up Next" horizontal strip along the bottom (`queue-items`, kb-x auto-scroll, click/Enter → `queue-jump(entry.index)`; chip titles word-wrap into a fixed 2-line-tall box instead of eliding to one line, so a long track title is fully readable). Transport row also carries 🔉/🔊 volume buttons (ctrl-focused 6/7, same `volume-up`/`volume-down` callbacks as the music bar and fullscreen player). Entry: click the music bar's art/title zone (`open-now-playing` callback — resets focus state then shows), the `m`/`M` key (toggles open/close from any mode), or auto-open after ~30s idle (`Config.now_playing_auto_open`, Settings → Audio row 12, default on — fixed threshold, not user-tunable in v1). Idle tracking: `VideoState.music_idle_ticks` incremented every 16ms tick while `current_is_audio && player.is_some()`, reset to 0 on every keypress (`on_handle_key` in `main.rs`) and pinned to 0 while the screen is open (so any close path needs a fresh 30s before it can re-fire); at exactly 1875 ticks (~30s) the timer sets a local `auto_open_now_playing` flag and calls `invoke_open_now_playing()` only AFTER releasing the `VideoState` lock (same deferred pattern as `gapless_commit`/`banner_trigger` in this function) — calling it while still locked self-deadlocked the UI thread, since the callback's `refresh-queue-display` handler re-locks the same mutex (mpv's own audio thread kept playing regardless, matching the "music plays, UI frozen solid" symptom; fixed same day, verified via `coredumpctl` on a real SIGABRT). `on_open_now_playing` also calls `invoke_refresh_queue_display()` on every open (all three entry paths funnel through it) — the Up Next strip reads the same `queue-items` model as the Queue Panel, and this is still the only guaranteed *first* art fetch for it (`spawn_queue_poster_loading`, `main.rs`). `push_queue_display` itself (`main.rs`, called from ~13 sites — nearly every queue/playlist mutation, including every natural track advance) rebuilds `queue-items` from scratch every time, but preserves already-known art by `poster_id` (snapshotted from the model it's about to replace) rather than defaulting every row back to a blank placeholder — since same-album tracks share one `poster_id` (`album_art_id`), this alone covers a same-album advance for free. The two natural-advance sites in `playback.rs` (gapless commit and the non-gapless playlist/queue-advance branch) additionally call `spawn_queue_poster_loading` right after `push_queue_display`, so a genuinely new `poster_id` (the queue crossing into a different album) still gets fetched immediately rather than waiting for the user to manually reopen Now Playing or the Queue Panel — real bug, found live: before this, every mutation *other than* `on_refresh_queue_display` wiped the strip back to placeholders, which is why it "worked for track 1" (whichever handler most recently opened Now Playing) and then went blank on the very next advance. — and `invoke_grab_keyboard_focus()` (the global `fs` FocusScope can lose focus during a stretch of mouse-only interaction, most reliably before idle auto-open; every other screen-open site — season/person/detail/series/auth, and since CR11-3 also artist/collection/album, which were missing it — already re-grabs unconditionally for the same reason). The music bar's ⋮ Queue button has the analogous fix: `AppState.open-queue-panel()` (CR11-6) re-grabs focus too, since opening the Queue Panel by mouse has the same failure mode. Keyboard state: `now-playing-back-focused` (top bar), `now-playing-in-strip` (false=transport row focused, true=Up Next strip focused), `now-playing-ctrl-focused` (0=Open Album 1=⏮ 2=⏸/▶ 3=⏭ 4=⇌ 5=↺ 6=🔉 7=🔊), `now-playing-strip-focused` (index into `queue-items`) — Back-focused intercepts first (Enter/Back closes, Down → transport row); otherwise Up from the transport row → Back, Up from the strip → transport row (one level at a time, matching album/artist), Down mirrors it, Left/Right move within the active row, Enter activates (ctrl 0 is the only site that still calls `music-bar-open-album` — it opens the album screen and closes Now Playing; 6/7 call `volume-down`/`volume-up`; strip jumps via `queue-jump`). PrevTrack/NextTrack/ToggleShuffle/CycleRepeat/ToggleLyrics/Space-pause all fire via the existing global pre-dispatch before this mode's own handler ever sees them. Back/Esc anywhere closes; `reset_playback_ui` and sign-out also clear `show-now-playing`. The music-bar-focused and mini-player-bar-focused pre-dispatch blocks in `handle_key()` run BEFORE the per-screen `match mode` and originally only excluded `Player`/`ContextMenu`(/`QueuePanel`) — so a stale `music-bar-focused >= 0` or `float-card-focused >= 0` left over from before Now Playing opened silently hijacked its arrow keys/Enter. Both now also exclude `AppMode::NowPlaying`; the mini-player-bar-focused block was additionally missing `AppMode::QueuePanel` in its own exclusion list even though the music-bar-focused block already had it (CR11-4), and the global `ResumePlayer` ('r') pre-dispatch was missing both `QueuePanel` and `NowPlaying` (CR11-5).

**Bar layout**: content fills `y=0 height=parent.height-total-bar-h`; `total-bar-h = bar-h + music-bar-h` (72px when music bar active, else 0px). `MiniPlayerBar` sits at `y=parent.height-bar-h-music-bar-h`; `MusicPlayerBar` at `y=parent.height-music-bar-h`. All overlay screens and the loading overlay use the same `total-bar-h` padding.

The "Video in background" setting (persisted) controls whether Back during playback enters mode 2 or mode 3.

Settings → General row 2, "Log level" (`error`/`warn`/`info`/`debug`, `Config.log_level`, default `info`): governs the verbosity of both the console and `~/.cache/fjord/fjord.log`. Read directly off disk in `main()` before the `tracing` subscriber is constructed (the full `Config` isn't loaded into `FjordState`/`AppState` until later, after `window`/`state` exist), so a change takes effect on the next launch, not live. `RUST_LOG` still overrides it when set, for dev runs. Most of the per-card-model diagnostic logging added while chasing the Phases 96-100 library-grid flash bugs (`apply_cards_preserving_identity`, `spawn_poster_loading`'s per-section timing, `push_library_cards`, `refresh_library_display`, `update_library_filter`) lives at `debug!` — high-frequency and only useful when actively troubleshooting a similar issue; this setting is what makes them visible again without a rebuild.

Settings → General rows 3-4, "Prewarm library metadata" / "Prewarm library images" (Phase 104, `prewarm.rs`): button-style rows (like Sign Out), each triggering an opt-in, re-runnable, one-time-per-press full-library sweep — see the `prewarm.rs` module-table entry above for what each does. Both are a genuinely large one-time cost for a big library (a real run: metadata prewarm made 11,731 requests over ~8 minutes, 90% of them `get_person_filmography` — one per unique credited person, a reverse-lookup query pattern that's inherently more expensive server-side than a direct item fetch regardless of Jellyfin's own metadata caching) — this is why the log line exists: the user should be able to see the actual cost, not take it on faith. Deliberately **not** run automatically on login or on a timer; the ambient Phase 103 background refresh (`spawn_screen_cache_refresh`) that *does* run on every login is capped to its 40 most-recently-used entries per cache regardless of how large prewarm has grown `cap` (`BoundedCache::recent_keys` + `main.rs::AMBIENT_REFRESH_LIMIT`) — without that cap, a prewarmed library would repeat the prewarm's full request volume on every single login instead of once, a real bug caught via live testing before this shipped.

### Dashboards and library grid

There are four dashboard screens (horizontal `SectionRow` card rows) and one library grid:

- **Home dashboard** (`HomeScreen`, `active-nav == 0`, up to 5 rows): Continue Watching, Next Up, Recently Added Shows (`Series`), Recently Added Movies, Recently Added Music (albums). Shows movies, series, and albums.
- **TV Shows dashboard** (`DashboardScreen`, `active-nav == 1`, up to 5 rows): Continue Watching TV, Next Up, Recently Added Shows (`Series`), Not Watched Shows (`Series`), Favorite Series.
- **Movies dashboard** (`DashboardScreen`, `active-nav == 2`, up to 4 rows): Continue Watching Movies, Recently Added Movies, Not Watched Movies, Favorite Movies.
- **Collections dashboard** (`CollectionsDashboard`, `active-nav == 3`, up to 2 rows): Recently Added collections, Unwatched collections. Fetched as part of `fetch_home_data` (sections 9+10 in the 17-element poster-loading array). Enter on a card calls `on_item_play` → `open_collection_screen`. `on_item_play` checks `FjordState.all_collections` first (lazy, only populated when the library grid is opened), then falls back to the always-present dashboard Slint models (`recently-added-collections` / `unwatched-collections`) to obtain the BoxSet id+name — so the card works immediately on startup before the library grid is ever opened. Double-click the Collections sidebar item (or press Enter from the dashboard) opens the `LibraryGrid` with all BoxSets.
- **Music dashboard** (`MusicDashboard`, `active-nav == 4 && !show-library`, up to 4 rows): Recently Added Albums (`get_latest_music()` — music Latest needs a ParentId), Recently Played Albums (`get_recently_played_albums()` — played Audio tracks mapped to parent albums), Favorite Albums, Playlists (section indices 0-3; `music-playlists` model). Fetched as part of `fetch_home_data` (sections 11+12+15+16 in the 17-element array). Enter on a card routes via `on_item_play` → fetches item detail → `MusicAlbum` opens `album::open_album_screen`, `Playlist` opens `album::open_playlist_screen`. Individual `Audio` tracks play via `start_playback`; album art is fetched in background (generation-guarded) and shown in the music bar.
- **Library grid** (`LibraryGrid`, `show-library == true`): full poster grid of every item in a category. Opened by pressing Enter from the Movies, TV, or Music sidebar tab, or by double-clicking Collections. Separate concept from the dashboards — do not call this a "dashboard." For Music (nav=4) the sort bar layout is [Artists(0)][Albums(1)][Playlists(2)] | Sort: [A→Z(3)][Z→A(4)][Year↓(5)][Year↑(6)][Shuffle(7)] | [Favorites(8)]. View pills are at cursor 0-1 (left of sort pills, reached by pressing Left); Favorites is at cursor 7 (right of sort pills). `library-has-filters=false` for Music — the Favorites pill is rendered separately. All pills require Enter to apply; cursor starts on the active sort pill (at offset 3 for Music). Toggle pills require Enter to apply — cursor navigates freely without switching view. `library-music-view` (0=Artists, 1=Albums, 2=Playlists) persisted in `Config.library_music_view`; always one selected (radio semantics). Artists view: Enter on a card opens `ArtistScreen`. Albums view: Enter on a card opens `AlbumScreen` directly. Playlists view: Enter opens the playlist screen (`album::open_playlist_screen` — AlbumScreen in playlist mode); cards show "N tracks" subtitles (`ChildCount`); per-view sort in `Config.library_playlists_sort`.

Episode cards in dashboard rows display the series poster (`series_id` used as the fetch key), not the episode thumbnail. `spawn_poster_loading` carries a `poster_id` field alongside `item_id` in its metadata tuple for exactly this reason.

Not Watched rows use `SortBy=Random` so each fetch returns a different selection. A 30-second polling timer (`timer_nw`) refreshes the Not Watched row when the relevant tab is visible, no playback is active, and 10 minutes have elapsed since the last refresh. Timestamps `last_nw_mov_refresh` / `last_nw_tv_refresh` in `AppState` track this independently per tab.

Poster images are cached to `~/.cache/fjord/posters/` and decoded off the UI thread — JPEG decode runs on a Tokio worker producing `SharedPixelBuffer<Rgba8Pixel>` (which is `Send`), then `Image::from_rgba8` is called inside `invoke_from_event_loop` because `slint::Image` is `!Send`. Decodes are downscaled to their display size (`decode_poster_buffer` ≤600 px longest side; `decode_backdrop_buffer` ≤3840 px) — full-resolution decodes multiplied model memory ~6-10× for no visible gain.

`HomeItem` (defined in `theme.slint`) carries `has-played: bool`, `resume-pct: float`, `unplayed-count: int`, and `is-favorite: bool` — populated from `UserData`. Cards use two Jellyfin-style text rows via `CardItem.title`/`CardItem.subtitle` (`MediaItem::card_title()`/`card_subtitle()`): episodes show series name + "S1:E3 - Title", series show "2019 - Present" (Status/EndDate), albums show the artist, movies the year; inside series/season screens episode cards show episode name + "S1:E3". Both text rows wrap to as many lines as they need — no truncation — with the poster shrinking to make room (see the `vertical-alignment: bottom` Slint gotcha below for why this isn't a fixed-height box). `MediaCard` renders:
- **✓ badge** (top-right, accent circle, bold) when `has-played`
- **progress bar** (bottom) when `resume-pct > 0 && !has-played`
- **unplayed-count pill** (top-right, accent circle, bold) when `unplayed-count > 0 && !has-played` (series posters only)
- **♥ badge** (top-left, accent circle) when `is-favorite`

Card dimensions are computed by breakpoint pure functions (`dash-card-w`, `dash-card-h`, `grid-cols`) that live on `MainWindow` because they reference `self.width`. A `sync-layout()` function pushes the results to `AppState.dash-cw`, `AppState.dash-ch`, and `AppState.library-cols` on `init` and `changed width` so all screens see the current sizes.

### Disk caches
- `~/.cache/fjord/home.json` — home row data. Shown from cache immediately on warm start, always refreshed in the background.
- `~/.cache/fjord/movies.json` — full movie list (`Vec<MediaItem>`). Populated after first network fetch; on warm start loaded for instant display **without** setting `movies_fetched` — the first grid open each session still does a background network refresh (the flag only guards against multiple refreshes within one session).
- `~/.cache/fjord/series.json` — full series list. Populated at login/auto-login and on every background refresh. Loaded on warm start so Browse All and the TV grid are instant.
- `~/.cache/fjord/collections.json` — full BoxSet list (`Vec<MediaItem>`). Populated on first Collections library grid open; on warm start loaded immediately so the Collections grid shows content at once. Refreshed once per session on grid open (`collections_fetched` flag guards re-fetch).
- `~/.cache/fjord/artists.json` — full album-artist list (`Vec<MediaItem>`). Populated on first Music library grid open; on warm start loaded immediately. Refreshed once per session (`artists_fetched` flag guards re-fetch).
- `~/.cache/fjord/albums.json` — full MusicAlbum list (`Vec<MediaItem>`). Populated on first Music library grid open (Albums view); on warm start loaded immediately. Refreshed once per session (`albums_fetched` flag guards re-fetch).
- `~/.cache/fjord/playlists.json` — audio Playlist list (`Vec<MediaItem>`, video playlists filtered by `MediaType`). Same lifecycle as albums (`playlists_fetched`); additionally refreshed by `refresh_playlists` after every add-to-playlist / create.
- `~/.cache/fjord/posters/<id>` — raw poster bytes, one file per item, plus a `<id>.tag` sidecar holding the server's `ImageTags.Primary` hash. Grid/row loaders pass the item's current tag to `fetch_poster_cached_tagged`; a mismatch re-downloads the image (replaced artwork propagates). Callers that don't know the tag (screen-open fetches that run parallel to the detail fetch, person portraits, queue thumbnails) use the untagged path and serve whatever is on disk. A failed re-download falls back to the stale copy.
- `~/.cache/fjord/backdrops/<id>` — raw backdrop bytes + `.tag` sidecar (`BackdropImageTags[0]`), same revalidation rules.
- `~/.cache/fjord/last_cleanup` — Unix timestamp (ASCII) of the last poster/backdrop cache cleanup run.
- `~/.cache/fjord/screen_caches.json` — the six screen-open caches (Phase 103, see `config.rs`/`prewarm.rs` module-table entries), one file since they're always loaded/saved as a unit. Loaded on warm start (`push_cached_data`); saved every 60s by a repeating timer (`main.rs::wire_screen_cache_save_timer`) regardless of whether anything changed since the last write (the write itself is cheap). Can grow well beyond its default ~40-entries-per-cache footprint if the opt-in library prewarm (Phase 104) has been run — 1.3MB observed for a library with 10,609 unique credited people.
- `~/.cache/fjord/discover_posters/<mediaType>-<tmdbId>[-bg]` — Seerr/TMDB poster and backdrop bytes (`discover.rs`). Kept separate from `posters/`/`backdrops/` since TMDB has no Jellyfin-style `ImageTags` to revalidate against; permanent, no cleanup sweep (the library is small — only ever whatever's been searched for).

**Poster cache cleanup** (`run_poster_cache_cleanup` in `home.rs`): spawned as a background task after each auto-login. Takes `(movie_ids, series_ids, collection_ids, artist_ids, album_ids, playlist_ids, detail_ids)`. Builds a known-ID set from `all_movies ∪ all_series ∪ all_collections ∪ all_artists ∪ all_albums ∪ all_playlists ∪ detail_ids`, then walks `posters/` and `backdrops/` deleting any file whose name (= item ID, after stripping `.tag`/`.tmp` suffixes) is not in the set. `detail_ids` = `item_detail_cache` keys + every cached item's credited person ids — **cast portraits are cached under person ids, which the six flat lists never contain**; before this was added (2026-07-10) every 24h sweep deleted the portraits the image prewarm (Phase 104) and ordinary cast-row browsing had cached (8,346 files wiped in one observed run — the prewarm and the cleanup were silently fighting each other). Guarded by: (1) combined flat-library ID set non-empty (skips on network error / first run; `detail_ids` alone doesn't satisfy the guard since the persisted screen-cache file can be non-empty when the library fetch failed); (2) 24 h minimum interval via `last_cleanup`. Season/episode cache files still fall outside the known set and are re-fetched on next access.

On a warm start (valid saved session + fresh cache) the window opens on `ConnectingScreen` while the `check_auth` probe (8s timeout) confirms the server is reachable; once confirmed, cached content is pushed and the dashboard appears immediately (no loading flash from that point on) while a full background refresh runs. Previously cached content was pushed unconditionally before the probe ran, so a fully unreachable server looked identical to normal operation — see the Session Identity section above and the connection-issues review this decision came from.

### Keyboard navigation
A global zero-size `FocusScope` (`fs`) captures all keyboard input. `invoke_grab_keyboard_focus()` is called from Rust at startup **and after every login** (manual + auto-login) to give `fs` focus — without the post-login call, all keyboard navigation is dead until restart.

All keyboard input flows into `keys::handle_key()` in Rust (called from Slint's `key-pressed` handler). `active_mode()` derives the current `AppMode` (13 variants) from `AppState` flags — screen priority is encoded once in this function, not scattered across conditionals. A `match mode { ... }` then routes to per-module handlers: `context_menu::handle_key`, `person::handle_key`, `season::handle_key`, `series::handle_key`, `detail::handle_key`, `artist::handle_key`, `collection::handle_key`, `album::handle_key`, `dispatch_player`, `dispatch_library`, `browse::handle_key`, `dispatch_settings`, `dispatch_dashboard`. A pre-match check for `ResumePlayer` fires globally for all modes except Player/Person/Season/Series/Detail/Artist/Collection/Album/ContextMenu. `handle_global_shortcuts` (F/Ctrl+Q/B/1/2/3/S) is called as a fallback from both Dashboard and Settings arms. The contract is uniform: **Enter/Right enter**, **Backspace/Escape go back**, **Up/Down navigate rows/items**, **Left/Right navigate within a row or cycle a combobox**.

All keyboard state lives in the `AppState` global singleton. Key nav state:
- **`-1` = sidebar**: Up/Down cycle nav tabs (0 Home → 1 TV Shows → 2 Movies → 3 Collections → 4 Music → 5 Browse All → [6 Discover, only when `Config.seerr_enabled` — otherwise skipped entirely, not just hidden] → 10 Settings → 11 Quit → wrap); arrowing to nav=5 opens `show-browse` immediately; Right/Enter enters the content grid or library; `settings-focused` is reset to -1 when `active-nav` changes and also when `B` opens browse.
- **Discover** (`active-nav == 6`, `AppMode::Discover`, `discover.slint`): same dashboard tier as Home/Movies/TV, and — despite having its own `AppMode` rather than reusing `Dashboard` — the same `focused-section` sidebar/content split those get for free from `dispatch_dashboard` (`discover::handle_key` replicates it manually; see the Seerr integration section's bug write-up below for why this needed a live fix). `fs < 0`: Up/Down cycle sidebar tabs, Right enters the grid (or the search field, if there's nothing to focus). `fs >= 0`: 2D grid nav reuses `AppState.library-cols`; Up from row 0 focuses the search field (`discover-header-focused`, raw-key pre-dispatch `handle_discover_search` in `keys.rs`, mirrors `handle_browse_search`); Left-at-column-0 and Back return to the sidebar. Enter on a card opens **Request detail** (`show-request-detail`, `AppMode::RequestDetail`, `request_detail.slint`): Back-button-focus → `request-detail-zone` (0=button row, 1=storyline, 2=cast row — `existing_zones()` in `discover.rs` skips whichever zones don't apply to the current item; Up/Down step to the nearest existing zone, Enter activates per-zone). The Request button opens the **Request Options modal** (`show-request-options`, `AppMode::RequestOptions`, `RequestOptionsOverlay`) instead of submitting directly — Quality (2K/4K), quality profile chips, tag chips, season cards, and a Cancel/Request confirm row live there, navigated via `request-options-zone` (0=Quality row, 1=profile row, 2=tags row, 3=seasons row, 4=confirm row — `existing_option_zones()`, same skip-absent-zones idiom). See "Seerr integration" below for the full design.
- **`≥ 0` = content grid**: focused-section is the row index, `focused-card` is the column. Up/Down move between rows (Up at row 0 stays in content); Left/Right move between cards; Enter plays; I opens detail/series screen.
- **Browse list** (`show-browse = true`, `active-nav == 5`): opens in sidebar mode (`current-item = -1`). Up/Down navigate the sidebar; Right or Enter enters the list (`current-item = 0`). In list mode: Up/Down navigate items; Up at item 0 focuses the search bar (`browse-header-focused = true`); Left returns to sidebar; `/` also jumps to search. Search bar focused (`browse-header-focused = true`): typing filters client-side; Backspace deletes (empty → back to list); Down/Enter moves into results; Escape clears query and unfocuses. Backspace/Escape in list/sidebar mode closes browse and resets `active-nav = 0` when exiting via the Browse All sidebar entry. `B` shortcut also opens browse without changing `active-nav`.
- **Library grid** (`show-library = true`): 2D arrow nav across the poster grid; Enter opens detail; Backspace/Escape closes. Layout: top bar (54px) → sort bar (40px) → search field (52px) → poster grid with alphabet scrubber on right edge. Sort (0=Name A-Z, 1=Name Z-A, 2=Year↓, 3=Year↑, 4=Random) persisted per library type in `Config.library_movies_sort`/`library_series_sort`; filters reset to false on open. `refresh_library_display` in `browse.rs` is the single function that rebuilds `library-display` + `library-alpha-offsets` from current sort/filter/query. **UI section — animation speed** (`Config.scroll_speed_pct`/`animation_speed_pct`, both default 100, rows 0-1, always visible, no virtual rows): two multipliers, not one blanket setting and not per-animation-instance control. `AppState.settings-scroll-speed`/`settings-animation-speed` (`float`, e.g. `1.0` = 100%) are derived from the `-pct` int properties in `apply_settings_to_window` (and inline in `settings.slint`'s own dropdown `selected(v)` handlers, for the mouse-click path) — every `animate` block in the app multiplies its literal `duration:` by one of these two floats directly (`150ms * AppState.settings-animation-speed`), so `0%` collapses every transition to an instant jump and `150%` slows everything down, while preserving each animation's original *relative* pacing (a 90ms press-pulse stays proportionally snappier than a 600ms flash at any speed) — confirmed `duration * float` arithmetic works via a standalone throwaway Slint crate before committing to the design. Scroll (`viewport-y`/`viewport-x` on every keyboard-scrollable `Flickable` — 22 sites) is kept separate from `animation-speed` (the other ~99 sites: press-pulse borders, fades, background/color, geometry) because it's a throughput property (how fast you can move through content) rather than decoration. Applied to all 121 real `animate` sites (`grep -c "animate "` across `crates/fjord-app/ui/*.slint` finds a few more hits that are comment-only mentions) via a one-off Python script matching `animate PROP { duration: Nms` anywhere in a line (not just at line start — `main.slint`'s 28 `fadeN` sites have other content before `animate` on the same line) and classifying by whether `viewport-y`/`viewport-x` appears in the animated-property list. **Design journey worth knowing if this area gets touched again**: this started as a single blanket multiplier, was expanded per user request to a 4-category "Customize" toggle (Press/Fade/Background/Geometry, opt-in, hidden by default) for real per-category control, then deliberately collapsed back to just these 2 settings — the user's own concern was that with 4+ independent dials, a later "why does this look different" becomes undebuggable since nothing tracks what was changed. The categories were technically sound and the code for them was scoped out, just not built, in favor of staying simple. **Timers are a separate, deliberately mixed bag**: `LoadingSpinner`'s dot-cycle `Timer` (`widgets.slint`) directly choreographs the spinner's own rhythm alongside its dots' `animate opacity`, so its `interval` scales too — floored at `max(16ms, 260ms * settings-animation-speed)`, since unlike an `animate` block's `duration` (where `0ms` just means "jump instantly"), a `Timer` with `interval: 0ms` would re-fire continuously and spin the CPU. The other 4 `Timer`s in the app (`PressPulse`'s 110ms hold, `FadeInTrigger`'s 16ms kick-off tick, `PlayerScreen`'s 150ms pause/play flash hold, the toast auto-dismiss timer) are *not* scaled — they control how long a state is held before a separate, already-scaled `animate` block eases it away, which is a different property (patience, not motion). **Preset range** (`SPEED_PCT_MODEL` in `settings.rs`): `["0","25","50","75","100","150","200","300","400","500"]` — widened twice from the original `["0","50","75","100","150"]` after live testing, first to add `"25"`/`"200"`/`"300"`, then further to `"400"`/`"500"` on request. The percentage is a *duration* multiplier, not a rate, so a bigger number means the transition takes longer (slower), the opposite of what "speed" suggests; both row subtitles spell this out explicitly (`"... — lower % is faster, higher % is slower"`) to head off the same confusion. `current_value_str`/`apply_dropdown_selection`/`settings_row_action` in `settings.rs` all parse the selected string generically (`val.parse()`), so widening this array needed no other Rust-side changes — only `settings.slint`'s two dropdowns needed their `model:`/`current-value` ternary/`selected(v)` ternary extended to match, since those (the mouse-click path) hardcode the value set inline rather than referencing the Rust const. 500% is safe for both settings regardless of the Phase 123 `FadeGate`/layout finding above: the 11 AppShell-hosted sites no longer do delayed-unmount at all (reverted to instant unmount), and the 18 `FadeGate` sites are all free-floating overlays where a longer overlap is just a longer crossfade, not a layout hazard — so raising the ceiling doesn't reopen that regression. **Easing curve**: all 103 non-scroll `animate` sites use `easing: ease-out-expo` (`cubic-bezier(0.16,1,0.3,1)`) — changed from plain `ease-out` (`cubic-bezier(0,0,0.58,1)`) after live-testing at 300% speed: the user wanted the deceleration itself to feel more gradual, and `ease-out-quart` (one step up) wasn't a perceptible enough change, so it went straight to the most pronounced named curve (front-loaded speed, long lingering tail) rather than the more conservative option. The 14 scroll sites keep `ease-in-out` — a separate, untouched lever. Easing and the duration multiplier are independent properties on the same `animate` block; changing one doesn't affect the other.

**Screen/overlay exit fades — `FadeGate`** (`widgets.slint`): `main.slint`'s screen/overlay sites originally all used `FadeInTrigger`, which only handles the fade **in** — the moment the driving `show-X` flag flips false, Slint's `if` conditional destroys the element immediately, with no way for a plain `animate opacity` block to play on the way out (there's no element left to animate). This is why going Back from any of these screens felt instant despite entering them being animated — caught live when the user asked specifically about Detail-page Back navigation, but the root cause is generic to the `if`-gate pattern itself, not a Detail-specific bug. Fixed by adding `FadeGate`, which owns the mount/unmount decision instead of the raw condition: `gateN := FadeGate { show: <original if-condition>; }` declared as an always-present sibling (same "declare once, reference elsewhere in the component" pattern already used for `toast-timer`/`_toast-vis` in `main.slint`), then `if gateN.mounted: Element { opacity: gateN.fade-opacity; animate opacity {...} }` — `mounted` stays true for one fade-out's worth of time (`max(16ms, 150ms * settings-animation-speed)`, matching the consumer's own `animate` duration) after `show` goes false, so the element is still there to actually play the fade-out before the `if` unmounts it. (Property named `fade-opacity`, not `opacity` — every Slint element already has a built-in `opacity` property, so `FadeInTrigger` hit the same naming collision originally and same fix applies here.)

**Not safe everywhere, though — real regression, fixed same day.** Converting all 29 `main.slint` fade sites to `FadeGate` in one pass broke every dashboard/library/settings switch: two screens briefly mounted at once (outgoing fading out, incoming fading in) is fine for a free-floating overlay, but `AppShell` (`layout.slint`) `inherits HorizontalLayout` and its 11 content-slot children (BrowseScreen, HomeScreen, 2×DashboardScreen, 4×LibraryGrid, CollectionsDashboard, MusicDashboard, SettingsScreen) each carry `horizontal-stretch: 1` in that one shared slot, which assumes exactly one is mounted at any instant — with both temporarily present, the layout split/squeezed space between them instead of giving the whole content area to one, visibly breaking every grid/dashboard for the fade-out window (worse at higher `settings-animation-speed`, since that's a longer window). **Fix:** those 11 AppShell-hosted sites were reverted to plain `FadeInTrigger` + instant unmount (entrance-only, matching their original behavior) — a shared-layout slot needs its conditions to stay genuinely mutually exclusive at every instant, not just eventually, so exit-fade-via-delayed-unmount is fundamentally the wrong tool there. The remaining 18 sites keep `FadeGate`: ConnectingScreen/OfflineScreen/LoginScreen and all 7 overlay screens (series/season/artist/collection/album/detail/person) are free-floating direct children of `MainWindow` with explicit/absolute positioning and no enclosing `Layout`, as are now-playing/context-menu/playlist-picker/queue-panel/lyrics/volume-overlay/toast/the Up Next banner — none of them compete for shared layout space with a sibling, so simultaneous mount-during-fade is purely a visual crossfade, not a layout hazard. **General lesson:** before applying a "delay unmount until the fade-out finishes" pattern to an `if`-gated element, check whether it's a free-floating/absolutely-positioned child or a participant in a shared `Layout` (`HorizontalLayout`/`VerticalLayout` with `stretch`/flex sizing) — the pattern is only safe for the former; a Layout's per-instant mutual-exclusivity assumption is silently violated by the latter, and the failure mode (squeezed/split content) looks nothing like a fade-timing bug at first glance.

Both counts have grown since (Discover's `active-nav==6` AppShell content slot joined the `FadeInTrigger` group per this same rule; ConnectSeerrScreen and RequestDetailScreen joined the `FadeGate` group as genuinely free-floating overlays) — treat the two example lists above as illustrative of the *rule*, not a literal current count.

Both mechanisms scale with `settings-animation-speed`. `player.slint`'s 3 `FadeInTrigger` sites (skip-segment overlays, dismissed by explicit Skip/Don't-Skip button or countdown rather than a `show-X` flag flipping) and `MediaCard`'s poster fade-in remain on plain `FadeInTrigger` too — narrower, differently-shaped problems, out of scope for this pass either way.

**A second real `FadeGate` bug, found later the same week: LoginScreen visibly flashed to full opacity right as the window first appeared, on any cold start with a saved session.** Root cause was a timer-cancellation gap in `FadeGate` itself, not anything specific to Login. `AppState.show-login` defaults to `true` (before any config/auth-check has run), so `gate3`'s (LoginScreen's) `init` mounts it and arms its 16ms `kick-timer` immediately upon `MainWindow::new()`. Startup logic then flips `show-login` to `false` (and `show-connecting` to `true`) synchronously if a saved session exists — all of this happens *before* `window.run()` ever starts the event loop. `FadeGate`'s `changed show` handler's false-branch set `fade-opacity = 0` and started `unmount-timer`, but never stopped the already-armed `kick-timer` — so once the event loop finally started pumping timers, that stale `kick-timer` fired anyway and unconditionally set `fade-opacity` back to `1` (`kick-timer`'s `triggered` callback has no guard against `show` having changed since it was armed), flashing LoginScreen to full opacity on top of ConnectingScreen for roughly one fade cycle before `unmount-timer` finally caught up and unmounted it. **Fix:** the false-branch now also sets `kick-timer.running = false`, cancelling any pending fade-in kick the moment `show` flips false. This is a genuinely general bug — any `FadeGate` instance can hit the identical race if its `show` condition flips true→false within the same 16ms window, not just at startup — so the fix lives in `FadeGate` itself rather than as a special case for Login/Connecting. Root-caused via an Explore agent tracing the exact `main.rs` startup sequence (`MainWindow::new()` → `apply_settings_to_window()` → saved-session check → `window.run()`) against `FadeGate`'s own timer logic, not by guessing from the symptom alone.

### Press/click animation (Phase 105)
Every discrete clickable element (buttons, poster cards, menu/list/settings rows, music-bar icons, pills, tabs) shows a brief **border-flash** on activation — feedback that a click/Enter press actually registered, working identically for mouse and keyboard. Deliberately border-only, not a scale or opacity change: an earlier scale-down version visibly shrank-then-grew the whole widget including its child `Text`, which read as the text itself getting bigger/bolder/brighter on press — user feedback rejected that, so the mechanism only ever animates `border-width`/`border-color`, leaving text completely untouched.

**Mouse** needs no Rust involvement: every site binds `border-width: (ta.pressed || pulse.pulsing) ? W2 : W1` off Slint's native `TouchArea.pressed`. **Keyboard** has no equivalent "held" state (Enter is instantaneous, and `KeyEvent.repeat` is unreliable — see Slint gotchas), so it's synthetic: `AppState.kb-activate-pulse: int` is bumped once, centrally, in `keys::handle_key()` on every `key::RETURN` press (right after `mode` is computed) — every mounted `widgets.slint::PressPulse` instance (`trigger: AppState.kb-activate-pulse; active: <existing focus/selection expr>; out property pulsing: bool`, 110ms self-resetting `Timer`, same idiom as `FadeInTrigger`) reacts to the bump only if its own `active` input is true at that moment, so no per-widget identity scheme is needed — each site reuses whatever focus/selection boolean it already computes for its own hover/focus ring. Values: widgets with an existing 2px focus ring bump to 4px on press; widgets with no ring bump 0px→3px; a few special cases (`ToggleSwitch`, `SettingsDropdown`, the player's skip-segment prompt, the library-grid alphabet scrubber) are sized proportionally. Elements with no border-capable container (`QueuePanel`'s "Clear All", `album.slint`'s ✕ remove-track button — both bare `Text`, and Slint's `border-width` only exists on `Rectangle`) get a small underline/wrapping-box treatment instead of touching the `Text` directly. Excluded on purpose: drag-driven seek/progress/volume bars, the fullscreen player's click-to-pause overlay and cursor-hider `TouchArea`, backdrop-dismiss `TouchArea`s, and text input fields — none of these are discrete "click" actions.

`AppState.sidebar-kb-active: bool` (`app_state.slint`) is a second, narrower flag — true only under the same condition `keys.rs::active_mode()` calls `AppMode::Dashboard`, plus `focused-section < 0` — that exists to correctly gate the sidebar `NavItem`s' press-pulse (and the bespoke Quit row's press-pulse in `layout.slint`, which has the same shape bug independently since it isn't built from `NavItem`). It replaced `NavItem`'s original per-instance `kbd-focused` input prop entirely (removed, along with the identical `kbd-focused: AppState.focused-section < 0;` line every one of the 7 `layout.slint` call sites used to pass) — that old prop was a coarse, GLOBAL "sidebar has keyboard focus" proxy, bound identically on every sidebar row, that stays stale/true while deep in Library/Browse/Settings/Detail/Series/etc since those modes never touch `focused-section` at all. Deliberately implemented as a **pure Slint expression** (`out property`), not a value Rust pushes on keypress — mirroring `active_mode()`'s own condition chain directly off the same `AppState` flags (`show-context-menu`, `show-library`, `is-playing`, `focused-section`, etc.), at the cost of duplicating that condition list in two places (a screen/overlay added to one needs the matching branch added to the other). An earlier version *did* have Rust push it from inside `handle_key()`, refreshed every keystroke rather than gated to Enter — correct for keyboard-driven changes, but it went stale the moment a screen change was triggered by the *mouse* instead (clicking into the library grid never runs through `handle_key()` at all), so the sidebar's highlight kept reflecting whatever the last keypress had left it at. The pure-expression form is correct-by-construction for any input method, since the underlying `AppState` flags are set identically regardless of what triggered the change. Five real bugs traced to the original stale `kbd-focused` proxy, found via live testing in sequence: (1) gating the pulse on the old prop alone made every sidebar row flash simultaneously on Enter, not just the active one — fixed by requiring `active &&` together with it; (2) that still left the active tab pulsing on every unrelated Enter press anywhere in the app once focus had moved elsewhere — the first `sidebar-kb-active` fix; (3) `NavItem`'s own background highlight (`active && kbd-focused ? Theme.accent : ...`) stayed fully lit by the same staleness, since only the pulse's gate had been fixed, not the background's — switched to `sidebar-kb-active` too; (4) that version of `sidebar-kb-active` itself went stale for mouse-driven navigation, since it was only recomputed inside `handle_key()` — fixed by making it a pure expression; (5) once correct, the two-tier bright-accent-vs-muted-accent background it drove made Browse All and Settings look inconsistent with Home/TV/Movies/Collections/Music — those two open a separate `AppMode` (`Browse`/`Settings`) the instant the sidebar cursor lands on them (Browse All auto-opens on arrival even before Enter; Settings is checked directly by `active-nav == 10` in `active_mode()`), so they immediately dropped to the muted tier, while the other five all share `AppMode::Dashboard` regardless of which is selected and so stayed bright merely from being cursor-highlighted — visibly different behavior for what looked like the same kind of row. Resolved (per user preference, since the Browse/Settings/Quit look was judged cleaner) by dropping the two-tier distinction entirely: `NavItem`'s `background` is now a single `active ? Theme.accent-muted : (hover ? ... : transparent)` for all sidebar items, no longer reading `sidebar-kb-active` at all — that property's only remaining consumer is the press-pulse gate.

### Bundled symbol fonts (glyph coverage consistency)
Fjord uses many non-ASCII Unicode symbol glyphs throughout the UI (⏮︎⏭︎⇌︎↺︎⋮︎🔉︎🔊︎✓︎♥︎★ and more, ~40 distinct codepoints — mostly from the Miscellaneous Technical, Mathematical Operators, and Miscellaneous Symbols/Pictographs blocks). Fjord doesn't set any explicit `font-family` and never did — it always relied entirely on whatever the host's fontconfig default stack happened to cover, which is fine on a desktop with a full font set installed but not guaranteed on every HTPC. Confirmed live: several music-bar icons (Prev/Next/Queue/Vol-down/Vol-up) rendered as tofu boxes on the HTPC while working fine on the dev machine — the dev machine's fuller desktop environment happened to have `NotoSansSymbols2`/`NotoSansMath`/even an incidental GNOME-dependency font (`AdwaitaSans`) covering different subsets of these glyphs, none of which are guaranteed present on a minimal HTPC install. Root-caused via `fc-query --format='%{charset}'` cmap inspection (not `fc-match`, which always returns *some* font even for a poor/no-glyph match and so can't be trusted to confirm real coverage) against every symbol codepoint actually used in `crates/fjord-app/ui/*.slint`.

Fixed by bundling two Apache-2.0-licensed fonts directly in the binary: `assets/fonts/NotoSansSymbols2-Regular.ttf` + `NotoSansMath-Regular.ttf` (license text alongside as `NOTO-FONTS-LICENSE.txt`), imported via `import "../../../assets/fonts/....ttf";` at the top of `main.slint`. This is Slint's own documented mechanism for exactly this purpose (see Slint's Font Handling docs) — any `import "*.ttf"` found anywhere in the compiled project is collected and registered as a glyph-coverage fallback for the *whole app*, automatically, with no `font-family` binding needed on any consuming `Text` element; confirmed the font bytes are genuinely embedded by grepping the compiled binary for a byte slice pulled directly from the source TTF files. Between them these two fonts cover every one of the ~40 non-ASCII codepoints used in the UI except two purely decorative ones in the `PlaylistPicker` (`＋` fullwidth plus, `▎` one-eighth block, both in `context_menu.slint`) — rather than pull in a third font for two minor glyphs, those were simply swapped for plain ASCII (`+`, `|`), which every font on every system covers by definition. Deliberately **not** Noto Color Emoji — per explicit user preference, no color glyphs anywhere in the app; both bundled fonts are monochrome outline/symbol fonts, matching the rest of the UI's existing icon style. Could not get a reliable live screenshot verification in the sandboxed dev environment this was implemented in (window-focus/compositor issues kept capturing unrelated desktop content); confidence here rests on the documented Slint mechanism plus direct cmap-level coverage verification, not a rendered screenshot — worth an explicit visual check on the HTPC before considering this fully closed.

**Primary text font — Inter default + user-selectable, plus a hard-pinned icon font (follow-up the same day).** Comparing candidate primary UI fonts (an HTML artifact rendering the same real Fjord content in each candidate) surfaced a related but distinct problem: several candidates — Adwaita Sans, Liberation Sans, Fantasque Sans Mono — turned out to have their *own* competing glyphs for common icons (♥ ✓ ▶ ★), confirmed via the same `fc-query` cmap technique. Left alone, a `primary-font, Noto-fallback` stack means icon appearance silently depends on which primary font wins the race for each glyph — the exact inconsistency the Noto bundling above was meant to eliminate, just moved up one level. Resolved with two rules, settled through live iteration with the user: **♥ and ✓ always render via bundled Adwaita Sans specifically** (the user's preferred style for those two, confirmed live against the comparison artifact); **every other icon glyph always renders via the bundled Noto Sans Symbols2** (falling through to Noto Sans Math automatically for anything Symbols2 itself lacks, same global-fallback mechanism as above). Neither is ever overridden by whichever primary font is active.

Implementation: `assets/fonts/Inter-Regular.ttf` + `Inter-Bold.ttf` (OFL-1.1, from Inter's official GitHub release — Fjord's titles use `font-weight: 700`, so bundling only Regular would make Slint synthesize a fake bold) and `AdwaitaSans-Regular.ttf` (OFL-1.1) imported the same way as the Noto fonts. `widgets.slint::IconCircleButton` (the single shared component behind every ♥/✓ toggle *and*, via `now_playing.slint`, the whole ⏮/▶︎-⏸︎/⏭/⇌/↺/🔉/🔊 transport row) computes its icon `Text`'s `font-family` from which glyph it's showing: `(root.icon == "❤︎" || root.icon == "✓︎") ? "Adwaita Sans" : "Noto Sans Symbols 2"` — one change correctly routes all ~16 call sites. `FjordButton` gained `icon-left`/`icon-right` properties (pinned to Noto, kept separate from `text`) specifically because **Noto Sans Symbols2 has zero Latin-alphabet coverage** (confirmed via the same cmap check) — a combined string like `"▶︎ Play"` given a single `font-family` override would leave the word falling through to an uncontrolled system fallback; every other standalone icon `Text` outside these two shared components (representative files: `context_menu.slint`, `browse.slint`, `album.slint`, `player.slint`, `settings.slint`'s passthrough warning) got the same treatment by hand, splitting any icon+prose mix into separate elements first. Two heart/checkmark usages outside `IconCircleButton` needed the identical per-glyph ternary (`MediaCard`'s favorite/played badges in `widgets.slint`, the played/favorite rows in `context_menu.slint` — the latter mixing a pinned glyph (✓/♥) with an unpinned one (○/♡) in the same ternary, so the `font-family` itself is a matching ternary rather than one static value).

Settings → UI gained a third row, **Text font** — same dynamic-dropdown shape as Settings → Audio → Audio output (`fetch_system_fonts()` in `main.rs`, mirroring `fetch_audio_devices()`: shells out to `fc-list : family` instead of `mpv --audio-device=help`, same startup-fetch/selected-callback/`settings.rs` special-casing pattern throughout). Lists "Inter (Fjord default)" and "System default" first, then every distinct font family installed on the host, alphabetically. `Config.ui_font_family` scheme: `"Inter"` = bundled default, `""` = system default (no override, today's original behavior), anything else = that system font's family name verbatim. Bound via `MainWindow.default-font-family` (not `font-family` — `Window` doesn't expose that property in Slint 1.16.1, only `WindowItem`'s `default-font-family`, which is exactly the right semantics anyway: every `Text` without its own explicit `font-family` picks this up, while the icon pins above keep their own more-specific binding regardless) — set **synchronously** from `Config` in `apply_settings_to_window` at launch so the app never renders with a placeholder font on the first frame, independent of how long the async `fc-list` enumeration takes to populate the dropdown's own display list.

### Session identity (DeviceId)

`JellyfinClient` carries a `device_id: String` field used in the `Authorization` header (`DeviceId="…"`). The internal `reqwest::Client` is built with a **30-second request timeout** so a server that accepts the TCP connection but stops responding never hangs the auto-login task or API calls indefinitely. On first run, `ensure_device_id()` reads `/proc/sys/kernel/random/uuid`, saves it to `~/.config/fjord/config.json`, and uses it for the lifetime of the install. This is critical: if two machines share the same DeviceId, Jellyfin invalidates one machine's token when the other authenticates, causing 401 errors on all API calls.

Sign-out clears only the auth fields (`server_url`/`user_id`/`token`) and re-saves config.json — `device_id` and all settings persist across sign-out.

On startup, after loading a saved session, `check_auth()` does a cheap `GET /Users/{id}/Items?Limit=0&Recursive=true` probe with its own **8-second timeout** (shorter than the client's 30s default — this is the one call the whole startup UI blocks on before showing anything). On 401 the login screen is shown. Any other failure (DNS/connect/timeout — genuinely can't reach the server) shows **OfflineScreen** instead of proceeding — cached content is no longer pushed to the UI until the probe succeeds, so a full outage is now visibly different from normal operation rather than looking like a quiet, stale dashboard. Three buttons (`offline-focused` 0/1/2, Left/Right + Enter): Retry (`AppState.retry-connection`), Change Server (`AppState.sign-out` — clears the saved session so a *permanent* failure, e.g. a decommissioned server, isn't a dead end with only a Retry button that can never succeed), Quit (`AppState.quit`). Ctrl+Q also quits directly regardless of focus, matching the global shortcut. On success, `push_cached_data` (instant display from `~/.cache/fjord/*.json`) runs before the normal background refresh (`fetch_home_data`/`get_all_series`/`get_system_info` + `ws::start_websocket`), all in `main.rs`'s `spawn_auto_login`, which the Retry button re-invokes with fresh clones. That background refresh always runs, even with a warm cache and nothing changed server-side — the WS can't retroactively report changes made while Fjord was closed — but applying its result uses `push_home_data_preserving_posters`/`refresh_row_preserving_posters` (not the plain `push_home_data`/`items_to_model` `push_cached_data` itself uses for the very first paint) so already-decoded posters from the cache push survive the repaint instead of flashing blank (Phase 93). Passwords are never stored — Jellyfin tokens don't expire under normal use.

**Secrets at rest.** Until the Seerr integration, `config.json` stored the Jellyfin `token` as plain JSON with no explicit file permissions — the only precedent when Seerr's own secrets (a session cookie or a static API key, see below) needed the same treatment. Rather than extend that plaintext precedent, `secrets.rs` encrypts all three (`token`/`seerr_api_key`/`seerr_session_cookie`) with AES-256-GCM, keyed via HKDF-SHA256 over the existing `device_id` (already persisted, stable per-install) plus a compiled-in app salt — deliberately **not** an OS keyring (Secret Service/KWallet/GNOME Keyring): a keyring dependency would behave differently on the dev machine (where one is normally running) than on a minimal HTPC install (where one may not be), exactly the class of "works here, silently doesn't there" bug this project has hit before with fontconfig glyph coverage. `config.rs`'s `load_config`/`save_config` are the only two functions that touch this — every other reader of `Config` still sees plain strings, encrypted only in the on-disk form. **Honest limit, not oversold:** the key is derived from `device_id`, which sits in plaintext in the *same file* — this stops the secrets from leaking piecemeal (a config.json pasted into a bug report, a naive grep for a known token pattern, an out-of-context backup), not a targeted attacker with the whole file and the (open-source, public) derivation. That's the right bar for a personal HTPC's threat model. **Migration:** `decrypt_field` tries real decryption first; on any failure (which is exactly what a pre-encryption plaintext value looks like — not valid ciphertext) it falls back to using the stored value as-is, so an existing install self-migrates to the encrypted form on its very next save rather than being silently signed out on upgrade. `save_config` also now chmods `config.json` to `0600` on every write (previously whatever the umask gave, typically `0644`) — defense in depth alongside the encryption, not a replacement for it.

### Seerr integration (Discover screen + media requests)

Fjord's stated long-term direction is a general media client, not just a Jellyfin viewer; Seerr (docs.seerr.dev, the unified successor to Overseerr/Jellyseerr, with native Jellyfin/Plex/Emby support) is the first step — a "Discover" sidebar tab (`active-nav == 6`, hidden entirely unless `Config.seerr_enabled`) to search for movies/TV not already in the Jellyfin library and request them. Full endpoint reference: `SEERR.md`.

**Auth — all 4 of Seerr's non-Plex sign-in methods, not just one.** `fjord_seerr::SeerrAuth` is a two-variant enum (`ApiKey(String)` | `Session(String)`, the latter the literal `"connect.sid=…"` cookie pair) — every authenticated `SeerrClient` request attaches whichever variant it holds as either an `X-Api-Key` or `Cookie` header, one branch point reused by every call. `ConnectSeerrScreen` (`connect_seerr.slint`) is a dedicated screen, not inline Settings rows — Settings has zero free-text-entry precedent anywhere else (every existing row is a toggle/dropdown/cycle value), and once the shape grew from "one API key field" to "URL + a 4-way method selector, three of which need their own username/password or email/password pair," a LoginScreen-style dedicated screen fit better than cramming that into a settings row. Reuses `LineEdit` (Slint stdlib, already proven in `login.slint`) rather than the app's own hand-rolled character-buffer input (`browse-search-append`-style) — this is a low-frequency setup flow, not a per-keystroke-latency-sensitive search field, so the native widget's own focus/cursor handling is the right tool, unlike Discover's own search field (see below). Jellyfin Quick Connect is genuinely passwordless: `initiate` returns a code shown large on screen, a 2s Slint `Timer` polls `check` until `{authenticated: true}`, then `authenticate` completes the handshake — the user never types anything into Fjord for that path. **Credential retention matches the existing Jellyfin philosophy exactly:** the username/password/email typed during setup are never persisted, only the resulting cookie or key.

**Config lifetime:** `seerr_enabled`/`seerr_url`/`seerr_auth_method`/`seerr_api_key`/`seerr_session_cookie` clear on Jellyfin sign-out, same bucket as `server_url`/`user_id`/`token` — a fresh sign-in needs Seerr reconnected too, by design (not treated as a device-level setting like `device_id`).

**Session expiry (401) is handled at the point of failure, not proactively.** `discover.rs::handle_seerr_error` checks `is_401` (downcasts the `anyhow::Error` back to `reqwest::Error` and reads `.status()`) only for session-auth clients (`SeerrClient::is_session_auth()`) — API keys don't expire, so a 401 there means a revoked/invalid key, surfaced as a plain error since reconnecting wouldn't help without a new one anyway. A session 401 calls `seerr_auth::clear_connection` (same function `Disconnect` uses) and toasts "Seerr session expired — reconnect in Settings," rather than forcing the user out of whatever they were doing.

**Discover results reuse `CardItem`/`MediaCard`, not a parallel card type** — confirmed all four existing badges (✓ played, resume-pct bar, unplayed-count pill, ♥ favorite) are `if`-guarded and render nothing when fed the zeroed/empty data every Discover-sourced `CardItem` has for those fields (they only ever mean something for Jellyfin items). One additive field, `availability: string` (`""` | `"requested"` | `"processing"` | `"partial"` | `"available"`), drives a 5th badge pill in `MediaCard` the same shape as the other four — every existing Jellyfin construction site leaves it at `Default::default()`'s `""`, zero risk to the ~25 other `CardItem` models in the app. `item-type` is `"DiscoverMovie"`/`"DiscoverTv"` (already a free-form string field, no schema change) so Enter-to-open dispatches correctly without a parallel type system.

**Posters are TMDB CDN, not Jellyfin** — the first time Fjord fetches images from anywhere but the user's own server (`https://image.tmdb.org/t/p/w500{posterPath}`, cached to `~/.cache/fjord/discover_posters/`, keyed `"<movie|tv>-<tmdbId>"`). `poster.rs`'s existing fetch functions are methods on the concrete `JellyfinClient` (not parameterized over base URL/auth), so `discover.rs` has its own small parallel pipeline instead — it reuses `decode_poster_buffer`/`decode_backdrop_buffer` verbatim (both already byte-agnostic) and the same two-phase pattern as everywhere else in the app: build plain Send-safe metadata off-thread, construct the actual `CardItem`/`slint::Image` only inside `invoke_from_event_loop` (`CardItem` always carries an `Image` field, so it's `!Send` regardless of whether that field is populated — the type itself fails `Send`, not the runtime value; a real compile error caught this during implementation, not foresight).

**Season requests: full per-season checkbox picker**, not "all seasons" only — `TvDetails.seasons` (TMDB-shaped: number/name/episode-count) with no per-season Jellyfin-availability field in Seerr's published API spec, so the picker is pure selection (default all-checked), not an availability display; `POST /request`'s `seasons` field gets the literal string `"all"` when every box is checked (matching the API's own shorthand) or an explicit number array otherwise. `fjord_seerr::SeasonsSelector` is an untagged-serialize enum for this.

**Keyboard integration follows existing patterns exactly, not new ones:** `AppMode::Discover`/`AppMode::RequestDetail` slot into `active_mode()`'s priority chain the same way `Album`/`Collection` do; Discover's 2D grid nav reuses `AppState.library-cols`' math verbatim (same shape as `dispatch_library`'s grid arm — Left/Right/Up/Down + row-tracking for scroll); Discover's search-field typing is a raw-key pre-dispatch in `keys.rs` (`handle_discover_search`) that bypasses the `Action`/`KeyMap` layer entirely before `active_mode()` is even computed, mirroring `handle_browse_search` — this is why `ConnectSeerrScreen`'s `LineEdit`s and Discover's own hand-rolled search field use two *different* keyboard-input mechanisms (native widget focus vs. raw-key pre-dispatch): they're solving different problems (low-frequency setup form vs. every-keystroke search-as-you-type consistency with the rest of the app), not an inconsistency.

**Three real bugs found live, all fixed the same session:**

1. **Keyboard navigation went dead everywhere after connecting to Seerr.** `ConnectSeerrScreen`'s `LineEdit`s hold genuine Slint keyboard focus while typing — Slint's actual focus, not the app's own global `fs` FocusScope that `AppState.handle-key()` is wired to. Closing the screen after a successful connect (`seerr_auth.rs::commit_connection`) never returned focus to `fs`, so every subsequent keypress anywhere in the app (not just Settings — Settings was just where it was noticed) silently went nowhere, with no error and nothing in the log. Same root cause as the documented post-login `grab-keyboard-focus` calls in `main.rs`'s auto-login path — any screen with a native-focus-stealing widget needs to explicitly re-grab `fs` when it closes, and this one didn't. Fixed at all three close paths: `commit_connection`'s success branch (`w.invoke_grab_keyboard_focus()`), the Escape handler in `keys.rs`, and the ✕/backdrop-click handlers in `connect_seerr.slint` (`AppState.refocus()`, the same callback `settings.slint`'s toggle/dropdown rows call defensively after every mouse interaction).
2. **No keyboard way out of the Discover tab whenever it had zero results** — which is every first visit, before a query is typed. Every other dashboard-tier tab (Home/Movies/TV/Collections/Music) gets its sidebar-vs-content keyboard split for free from `dispatch_dashboard`, which treats `AppState.focused-section` as the shared toggle (`< 0` = sidebar focused, `>= 0` = the screen's own content). Discover has its own `AppMode` (its content is a flat poster grid, not `dispatch_dashboard`'s `SectionRow` model, so it couldn't reuse that function directly) — but the original `discover::handle_key` never touched `focused-section` at all and only ever handled `Action::Up`, so Back/Left did nothing the moment `discover-results` was empty, which is the state the screen starts in. Fixed by having `discover::handle_key` replicate the same `focused-section` contract itself: `fs < 0` → Up/Down cycle sidebar tabs (`browse::sidebar_nav`, same function `dispatch_dashboard` calls) and Right enters the grid (or the search field, if there's nothing to focus yet); `fs >= 0` → the existing grid math, plus Left-at-column-0 and Back now return to the sidebar. `sidebar-kb-active`'s `active-nav != 6` exclusion (added when Discover was first wired up, before this fix existed) was removed accordingly — `focused-section < 0` alone is now sufficient, exactly as it already is for Home/Movies/TV. 3. **Search silently returned nothing against the real Seerr instance** — the very first thing that logging (from bug 2) actually caught: `error decoding response body: missing field 'media_type'`. Every `Deserialize` struct in `fjord-seerr/src/models.rs` used plain snake_case Rust field names (`media_type`, `poster_path`, `total_results`, `display_name`, ...) with no `#[serde(rename_all = "camelCase")]`, but Seerr's actual JSON is camelCase throughout (`mediaType`, `posterPath`, ...) — confirmed directly from the OpenAPI spec used to design this feature, but the mismatch was never caught before shipping since it was never tested against a live server. Serde requires an exact field-name match by default, so any *required* multi-word field (`SearchResult.media_type`) failed deserialization outright, loudly, the moment `/search` was actually called — but every equivalent field elsewhere that happened to be `Option<...>` with `#[serde(default)]` (`MovieDetails.poster_path`, `TvDetails.backdrop_path`, etc.) would have failed silently instead, just staying `None` forever even when the server sent real data, no error anywhere. Fixed by adding `#[serde(rename_all = "camelCase")]` to all 11 `Deserialize` structs in the crate — the general fix, not a per-field `#[serde(rename = "...")]` patch on just the one field that happened to surface loudly. (Request *bodies* were unaffected — `client.rs` builds those with `serde_json::json!({"mediaType": ..., ...})` using literal camelCase keys directly, not a derived struct.)

### Episode auto-advance
Behaviour depends on `Config.skip_credits_mode`:
- **`never-skip`**: no banner, no auto-advance — `banner_trigger` is never fired.
- **`always-skip`**: auto-advance immediately when credits position is reached (no banner shown); `start_playback` called directly from the timer via `invoke_from_event_loop`.
- **`ask`** (default): Up Next banner fires *during* playback at `VideoState.credits_start` (from the Intro Skipper `/Timestamps` response — `credits.start`) or when `duration >= 60 s AND duration - position <= 30 s` (fallback). `next_ep_banner_shown` flag prevents it firing more than once per episode. A configurable countdown (`Config.skip_credits_secs`, default 30 s, stored as `banner_trigger.2`) counts down; the banner auto-advances when it reaches zero. "Play Now" calls `on_play_next_ep`; "Skip" calls `cancel-auto-advance` which sets `next_ep_pending = None` and exits the countdown task without playing. Keyboard: `next-ep-banner-focused` (0=Play Now, 1=Skip), Left/Right toggle, Enter activates.

`banner_trigger` type is `Option<(String, Option<Arc<JellyfinClient>>, u32, bool)>` — the `u32` is countdown seconds (0 for always-skip, so loop body never executes), `bool` is `show_banner` (false for always-skip suppresses UI updates).

`next_ep_pending` lives in `VideoState` (not `FjordState`) so it is cleared atomically when a new video starts, preventing stale pending state bleeding across sessions.

Every `start_playback` call site must pass `series_id` so auto-advance works for plays from any screen. Audio call sites should pass `audio_meta = Some((artist, album_art_id))` so the music bar shows correct metadata; non-audio call sites pass `None`.

**Auto mark-played at the credits trigger** (2026-07-10): whenever the credits-trigger fires (same condition as the banner/auto-advance above — `skip_credits_mode != never-skip`), the current episode is explicitly marked played via `POST /Users/{userId}/PlayedItems/{itemId}` (`client.mark_played`), not left to Jellyfin's own position-based heuristic on the eventual `report_playback_stopped` call. This existed once before and was removed (`CR10-13`) because it stuck even when the user pressed Skip and kept watching from an earlier point; it's back with the missing half: `VideoState.credits_auto_marked_played` tracks that the auto-mark fired, and if playback position then drops back below the trigger point during the *same* episode — i.e. the user rewound after Skip — `client.mark_unplayed` fires (`DELETE .../PlayedItems/{itemId}`) and the flag resets, so a Skip-and-rewind self-corrects instead of leaving a stale played mark. Both calls are dispatched from `credits_mark_played: Option<(String, Arc<JellyfinClient>, bool)>`, captured inside the 16ms tick's `VideoState` lock and spawned after it releases (same deferred pattern as `banner_trigger`/`gapless_commit` — calling the client while still holding the lock would block the tick). Best-effort: a failed call just logs a warning, matching every other playback-reporting call in this file.

The trigger itself is `credits_fire || fallback_fire` (`credits_start` from Intro Skipper, or a `duration - 30s` fallback when that's unavailable) — whichever crosses first. **Bug fixed in review (2026-07-11):** the rewind-revert check originally recomputed its own threshold as `credits_start.unwrap_or(duration - 30s)`, always preferring `credits_start` when present — but for any episode whose end-credits run under ~29s, `fallback_fire` crosses *before* `credits_start` does, so the mark-played fires via the fallback while the revert-check's independently-recomputed threshold still pointed at the later `credits_start`. That made the revert fire on the very same tick as the mark, before the lock ever released — the `mark_played` POST was silently overwritten by a `mark_unplayed` DELETE before either ever left the function, and the whole feature quietly no-op'd (or worse, un-marked a genuine rewatch) for that entire class of episodes. Fixed by adding `VideoState.credits_mark_threshold: Option<f64>` — the *actual* position that fired the trigger (`min` of whichever of `credits_fire`/`fallback_fire` was true), set alongside `credits_auto_marked_played` and read back by the revert check instead of recomputing. General lesson: when two branches of an `A || B` trigger a shared side effect, any code that later needs to know "did this already un-happen" must remember which branch actually fired, not re-derive a threshold from state that can independently disagree with what fired the very check it's guarding.

**Bug found from a real HTPC log (2026-07-11):** even with the above fixed, a user's `fjord.log` showed the same symptom the whole feature was built to prevent — a genuinely-watched episode still showing up in Continue Watching. The WS trace made the mechanism visible directly: `mark_played` fires correctly at the credits trigger (`UserDataChanged` confirms `played=true position_ticks=0`), but every teardown path (stop/replaced/natural-end/quit) funnels through the single shared `tear_down_player` helper, which unconditionally reports the *raw* mpv position via `report_playback_stopped` — regardless of whether the item was just explicitly marked played. Moments later a second `UserDataChanged` arrives showing `played=true` but `position_ticks` nonzero again (2917s in the observed log, from a user who kept watching past the credits point for a few more seconds before pressing Stop). A nonzero resume position on the server is what makes Jellyfin's `IsResumable` filter — which `get_continue_watching()` queries with `Filters=IsResumable` — match again, independent of the `Played` flag; the ordinary stop-report silently undid the mark-played moments after it succeeded, on every single episode, not just short-credits ones. Fixed at the same single point: `tear_down_player` now reports `ticks=0` instead of the raw position whenever `credits_auto_marked_played` is still true (i.e. never reverted by a rewind), and clears both `credits_auto_marked_played`/`credits_mark_threshold` itself so the field doesn't need resetting at each of the 4 call sites individually. This complements the rewind-revert logic rather than replacing it: rewind past the trigger still flips `credits_auto_marked_played` back to `false` first, so a genuinely-abandoned rewatch still reports its real (resumable) position through this same path unchanged.

**Bug: last episode of a series auto-restarted itself at natural end (2026-07-11):** user report — no Up Next overlay shows for a series' last episode (correct), but letting it play to the literal end causes it to immediately restart from the beginning. Root cause: `/Shows/NextUp` (`get_next_up_for_series`) is unreliable right at an episode boundary in two ways, and this codebase only ever guarded against one of them. (1) Shortly before Jellyfin has processed a stop/played report for the current episode, NextUp still returns the CURRENT episode itself as "next up" — the pre-existing `if current_item_id == next.id` check in the banner-trigger path (CR10-13) resolves this by re-deriving the true next episode from the series' own ordered episode list. (2) Once a series is FULLY watched — which, after the credits-trigger auto-mark above, can now happen well before natural EOF for the very last episode — NextUp can fall back to a "start over"/rewatch suggestion (observed live: it suggested the just-finished episode itself) instead of correctly reporting "no next episode." The natural-end-of-file fallback path (`wire_mpv_timer`'s `if had_series { ... } else if let Some(sid) = advance_series_id { ... }` branch, which runs when EOF arrives before the banner-trigger's own background fetch has set `next_ep_pending`) called `get_next_up_for_series` directly with **no** such guard at all, so case (2) sailed straight through and restarted the show. Fixed by extracting both call sites' resolution into one shared `resolve_true_next_episode(cli, series_id, current_id)` helper (`playback.rs`, near `playlist_next`) that never calls `/Shows/NextUp` at all — it fetches the series' ordered episode list directly and only accepts a genuine current-position+1 match as "next"; anything else (same episode, an earlier one, any other NextUp-style suggestion) means there is no next episode, full stop. This is a general fix, not a second special case bolted onto the first — both the banner-trigger path and the natural-end fallback now share the identical, NextUp-independent resolution logic instead of each reinventing (or, in the fallback's case, omitting) its own guard.

**Two more mark-played bugs found in review (2026-07-11):** a code review covering everything since CR11 turned up two more ways the credits-trigger mark could be silently undone, both variations on the same root cause as the `tear_down_player` fix above — code written before this feature existed that reports the raw playback position with no awareness `credits_auto_marked_played` exists. (1) `report_playback_progress` (the routine ~10s-interval Jellyfin progress report that runs throughout ALL playback, not just at teardown) reported the live, non-zero position completely unconditionally — meaning even if `tear_down_player`'s fix works perfectly, the very next progress tick after the credits trigger (which can land anywhere from 0-10s later, since `pos_tick`'s modulo-600 cycle isn't reset by the trigger) re-adds a resume point and undoes the mark long before the user ever stops or reaches EOF. Fixed by adding the identical `&& !vs.credits_auto_marked_played` guard used at the trigger site. (2) The rewind-revert block correctly reset `credits_auto_marked_played`/`credits_mark_threshold` on a genuine rewind, but never reset `next_ep_banner_shown` — since the outer trigger block is gated on `!next_ep_banner_shown`, once it fires once for an episode it can never fire again, even after a legitimate rewind-and-rewatch. A user who rewound past the credits point to re-watch a scene, then watched forward through the same credits point a second time and stopped before literal mpv EOF, would end up with a genuinely-watched episode never re-marked played — the exact failure mode this whole feature was built to prevent, just reached through a different door. Fixed by resetting `next_ep_banner_shown = false` alongside the other two fields in the rewind-revert block; re-showing the Up Next banner on a second pass through the credits window is correct, expected behavior (the "once per episode" comment predates rewind-tracking entirely), not a regression.

**Other findings from the same review, fixed:** `layout.slint`'s Quit sidebar row had its press-pulse migrated to `sidebar-kb-active` (Phase 105) but its `background` binding was missed, left reading the stale `active-nav==11 && focused-section<0` check — arrowing to Quit then opening any overlay via mouse (which doesn't touch `active-nav`/`focused-section`) left the row lit red for as long as the overlay stayed open. Sign-out never cleared the six Phase 102/103 screen-open caches (`item_detail_cache` and friends) — since they're keyed only by item id with no user/server scoping, a second account signing into the same install would silently see the first account's cached played/favorite state on anything viewed before sign-out, self-healing only via an unrelated WS event or FIFO eviction; added `BoundedCache::clear()` and called it for all six on sign-out. `widgets.slint::PressPulse`'s `reset-timer.running = true` was a no-op when the timer was already running (Slint's `Timer.running` is dirty-checked like any property), so a second `kb-activate-pulse` bump within 110ms of the first didn't restart the countdown — confirmed against `i-slint-core`'s actual `Property::set`/Timer source, not assumed; fixed by forcing a genuine `false` then `true` write, both real transitions. `prewarm.rs::spawn_image_prewarm` still had a `keys()` + `get()` double-lookup the earlier `BoundedCache::iter()` cleanup (this same day) missed — switched to `iter()`, which made `keys()` itself genuinely dead and it was deleted.

**Three design-tradeoff findings from the same review, also acted on:** `resolve_true_next_episode` (above) originally dropped `/Shows/NextUp` entirely rather than validating it — simpler, but throws away NextUp's server-side watched-state awareness (e.g. correctly skipping an episode already watched from another client) for scenarios this codebase hadn't yet hit in testing. Changed to a validated-hint form: fetch the ordered episode list either way (still needed to know the current episode's position), but now also call `/Shows/NextUp` and accept its answer whenever it resolves to a position strictly greater than the current episode's — only falling back to strict position+1 when NextUp's answer fails that check (same id, an earlier episode, or anything else not verifiably forward). Both original failure modes are still caught by the same position check; the skip-ahead case is now preserved. `prewarm.rs::spawn_metadata_prewarm` (Phase 104) had no session guard on its multi-minute sweep, unlike `ws.rs`'s `Arc::ptr_eq` pattern (CR11-2) for the identical class of risk — added a matching `session_current()` helper, checked before every write that inserts a `MediaItem` (which embeds per-user `UserData`) into a shared cache, across both the two batched top-level/person-detail loops and all five of the JoinSet relationship-cache closures. Deliberately *not* added to `spawn_image_prewarm` — that function only caches raw image bytes keyed by item/person id, which aren't per-user, so a stale client there just fails the request (logged, harmless) rather than corrupting state. `push_cached_data` (`main.rs`) ran entirely on the Slint UI/event-loop thread (called from inside `invoke_from_event_loop`), including `load_screen_caches()`'s synchronous `std::fs::read_to_string` + JSON parse of `screen_caches.json` — a file that can reach ~1.3MB after a library prewarm. Moved that one load (the only one of `push_cached_data`'s several `load_X_cache()` calls actually flagged, and by a wide margin the largest on disk) to run via `tokio::task::spawn_blocking` *before* `invoke_from_event_loop` is entered, threading the already-parsed `Option<ScreenCachesFile>` into `push_cached_data` as a new parameter instead of having it call `load_screen_caches()` itself. The other, much smaller `load_X_cache()` calls in the same function were left as-is — not flagged, and restructuring all of them would have been a much larger change for comparatively little benefit.

**Follow-up review (2026-07-11): the same review's 5 finder agents that hit the session limit were relaunched once it reset, and found 8 more real issues, fixed the same day.** `session_current()` (promoted from `prewarm.rs` to `main.rs`, `prewarm.rs` now calls `crate::session_current`) was missing from `spawn_screen_cache_refresh` — the AMBIENT per-login refresh of the six screen-open caches, which runs automatically on every single login, not opt-in like the prewarm sweep it was originally added for — so the exact same cross-session cache-contamination risk existed there too, just less obviously since nobody has to click a button to trigger it. Added to all 6 of its write sites (1 batched fetch + 5 JoinSet closures), matching the pattern already used in `prewarm.rs`. `spawn_metadata_prewarm`'s own finalization block (clearing `prewarm_metadata_running`, writing the summary) wasn't gated either — a stale sweep's completion landing after a *new* session's sweep had already started would reset the new sweep's own reentrancy guard out from under it and overwrite its summary text; added the same guard there too.

The rewind-past-credits revert (Phase 109/114) reset `credits_auto_marked_played`/`credits_mark_threshold`/`next_ep_banner_shown` but never touched the in-flight Up Next banner itself: `VideoState.next_ep_pending` stayed `Some`, `show-next-ep-banner` stayed visible, and the already-spawned countdown task kept ticking down untouched. A user who rewound to re-watch a scene while the banner's countdown was still running would have the *original* countdown auto-advance to the next episode on its original schedule regardless of the revert — auto-playing the next episode out from under someone who was deliberately still watching the current one. Fixed by having the revert also clear `next_ep_pending` (the countdown task's own per-second loop already checks `pending_ok` and bails within ~1s when it's `None` — the same mechanism the existing "Skip" button's `on_cancel_auto_advance` handler already used) and signal the UI to hide the banner immediately via the same deferred-dispatch-after-lock-release pattern used elsewhere in this function.

Separately, suppressing `report_playback_progress` entirely (not just zeroing it) for the whole `credits_auto_marked_played` window meant the server's position stayed at 0 (set by the original `mark_played`) for as long as that window lasted. When a revert then fired `mark_unplayed` — which, mirroring `mark_played`'s own behavior, also resets position to 0 server-side — the resulting `UserDataChanged` event reported `played=false, position_ticks=0`, which `ws.rs`'s own handling reads as "untouched item" and drops from Continue Watching — misfiring during an active rewatch, since the user's real position is nonzero. Self-healing within ~10s (the next ordinary progress tick), but fixed properly: `credits_mark_played`'s dispatch now also sends one immediate `report_playback_progress` call with the real position right after a successful `mark_unplayed`, closing the window down to effectively zero instead of waiting for the next periodic tick.

Two Slint screens' keyboard paths return early in `keys.rs::handle_key()` — before the single centralized `kb-activate-pulse` bump on `RETURN` — so their `PressPulse`-driven widgets never flash on keyboard Enter, only mouse click: `OfflineScreen`'s three `FjordButton`s (Retry/Change Server/Quit) and `PlaylistPicker`'s rows. Added the same bump to both early-return branches. Two glyph ternaries from the Phase 105 U+FE0E consistency pass only got the variation selector on one branch (`"✓︎" : "○"` and `expanded ? "▼" : "▶︎"`) — fixed both to match on both branches, the same class of inconsistency that pass was meant to eliminate everywhere else.

`container_tracks_cache` (one of the six screen-open caches) wasn't invalidated by either playlist-mutation path that should have cleared it: removing a track (`on_playlist_remove_entry`) left the deleted track visible on reopening the same playlist, since `open_music_screen` skips its network fetch on a cache hit; and `refresh_playlists`'s own CR11-7 "reopen the currently-viewed playlist so its tracklist reflects the change" fix was silently defeated by the same cache — the reopen call just re-served the pre-mutation cached list. Fixed both by invalidating the entry for that playlist id before the reopen/re-render.

### Intro Skipper plugin
When starting playback of an Episode, `start_playback` spawns one background task:
- **All segments**: `client.get_episode_timestamps(item_id)` (`GET /Episode/{id}/Timestamps`). Returns `EpisodeTimestamps { introduction, credits, recap, preview, commercial }` where each is a `Segment { start: f64, end: f64 }`. Valid when `end > 0.0`. On success the matching `VideoState` fields are populated: `intro_timestamps`, `recap_timestamps`, `preview_timestamps`, `commercial_timestamps`, and `credits_start = Some(ts.credits.start)`.

Returns `None` gracefully when the plugin is absent (404).

The 16 ms timer checks the current playback position against each segment in priority order (Intro → Recap → Preview → Commercial). At most one segment is active at a time. Behaviour per segment depends on the configured mode (`settings-skip-*-mode` from `AppState`):
- **`always-skip`**: seek to segment end immediately, set `skip_segment_handled = true`, hide overlays.
- **`ask`**: show `show-skip-segment` (single "Skip →" button). Enter in `dispatch_player` calls `invoke_skip_segment()`.
- **`ask-timed`**: on first tick set `skip_timed_shown_at = Some(Instant::now())`. Each tick compute `remaining = prompt_secs - elapsed`. Set `show-skip-timed = true`, update countdown label. When remaining ≤ 0: seek to segment end, set `skip_segment_handled = true`, hide overlays. User can "Don't Skip" via the overlay button or Esc (calls `invoke_dismiss_skip_timed()`, sets `skip_segment_handled = true`, hides overlay — suppresses re-show while still in segment).
- **`never-skip`**: hide all overlays, do nothing.

`VideoState` fields: `skip_segment_handled: bool` — set after seek or dismiss, reset to false when position exits the segment; `skip_timed_shown_at: Option<Instant>` — when the timed overlay first appeared; `skip_timed_prompt_secs: u32` — snapshot of per-segment secs at overlay start. `show-skip-timed`, `skip-timed-label`, `skip-timed-secs`, `skip-timed-focused` in `AppState` drive the UI.

**Stale-response guard:** `VideoState.playback_generation` is a `u64` counter incremented at the top of every `start_playback` call. Each spawned task captures the current generation and discards its result if `vs.playback_generation` no longer matches when the response arrives. This prevents a slow network response for episode A from overwriting episode B's `intro_timestamps` or `credits_start` after a fast episode skip.

### Async strategy
Tokio for all async. The Slint event loop runs on the main thread. Background tasks (API calls, poster fetching) use `tokio::spawn`. Communication back to the UI uses `slint::invoke_from_event_loop` or channels.

## What is Jellyfin

Jellyfin is an open-source media server. It exposes a REST API for browsing libraries (movies, TV shows, music) and getting playback URLs. Auth is username+password → returns an API token that goes in every subsequent request header as `X-Emby-Token` (Jellyfin kept the Emby header name).

Key API endpoints used:
- `POST /Users/AuthenticateByName` — login
- `GET /Users/{userId}/Items` — browse/search items
- `GET /Users/{userId}/Items?Ids=a,b,c&Fields=…` — explicit id-list fetch (`get_items_by_ids`, chunked 200/req); response order not preserved. Used by the WS delta-sync path (Phase 89) to resolve `LibraryChanged`/`UserDataChanged` ids without a full list re-fetch.
- `GET /Users/{userId}/Items/{itemId}` — item detail (overview, cast, backdrop tags, etc.)
- `GET /Items/{itemId}/Images/Primary` — poster image
- `GET /Items/{itemId}/Images/Backdrop/0` — backdrop image
- `GET /Users/{userId}/Items?Filters=IsResumable` — continue watching
- `GET /Users/{userId}/Items/Latest?IncludeItemTypes=…&GroupItems=true` — "Latest Media" rows (new episodes grouped into series, played incl.; bare array response). **Music caveat:** returns nothing for `IncludeItemTypes=Audio` — must be scoped with `ParentId={music view id}` (resolve via `/Users/{id}/Views`, `CollectionType=="music"`); grouped tracks come back as MusicAlbum items. Recently-played likewise can't use `Filters=IsPlayed` on MusicAlbum (album = played only when ALL tracks are) — query played Audio tracks and map to parent albums.
- `GET /Shows/NextUp` — next unwatched episode per series (home row)
- `GET /Shows/NextUp?SeriesId=…` — next episode for a specific series (auto-advance)
- `GET /Shows/{seriesId}/Seasons` — season list
- `GET /Shows/{seriesId}/Episodes?seasonId=…` — episode list for a season
- `GET /Videos/{itemId}/stream?static=true&api_key=…` — direct-play URL
- `POST /Sessions/Playing` — report playback started
- `POST /Sessions/Playing/Progress` — report position
- `POST /Sessions/Playing/Stopped` — report stopped
- `GET /Episode/{itemId}/Timestamps` — all skippable segments (Introduction, Recap, Preview, Commercial, Credits) in one call (Intro Skipper plugin v2+, optional)
- `GET /Items/{itemId}/Similar?userId=…&Limit=12&Fields=…` — similar items (same type, movies or series)
- `GET /Users/{userId}/Items?IncludeItemTypes=BoxSet&Recursive=true&Fields=Id,Name` — all BoxSets (collection map build)
- `GET /Users/{userId}/Items?ParentId={boxsetId}&Fields=ProductionYear,UserData` — items in a BoxSet (collection row)
- `GET /Users/{userId}/Items?IncludeItemTypes=Playlist&Recursive=true` — all playlists (`MediaType` filters audio ones)
- `GET /Playlists/{id}/Items?UserId=…` — playlist entries in order (each carries `PlaylistItemId`)
- `POST /Playlists` — create (`{Name, Ids, UserId, MediaType:"Audio"}`); `POST /Playlists/{id}/Items?Ids=…` — append
- `DELETE /Playlists/{id}/Items?EntryIds=…` — remove entries (PlaylistItemIds, not item ids)

## Known platform issues

### NVIDIA legacy Wayland: NVDEC stride corruption
**Symptom:** Diagonal stripe artifact (raw YUV scan lines) when using hardware decoding (`nvdec`, `nvdec-copy`). Software decoding is clean.

**Root cause:** NVDEC aligns decoded frame rows to 256-byte boundaries (e.g., a 1920-pixel-wide video gets a 2048-byte stride). mpv uploads via `glTexSubImage2D` with `GL_UNPACK_ROW_LENGTH=2048`. The NVIDIA legacy EGL driver silently ignores `GL_UNPACK_ROW_LENGTH`, so GL reads each row 128 bytes too tight — each successive row is offset from the previous, producing the diagonal slant.

**Fix:** Set `vf=format=yuv420p` in Settings → Video. This adds a software format conversion step after NVDEC decodes the frame, producing tight-packed yuv420p output so `GL_UNPACK_ROW_LENGTH` is never needed. For 10-bit HDR use `format=yuv420p10le`. The `auto` option detects the active hwdec and bit depth at runtime and picks the right format. `hwdec-image-format` was tried first but has no effect on NVIDIA legacy EGL.

**AMD Vulkan:** `vulkan-copy` works correctly with no stride workaround needed.

### PlayerConfig fields (fjord-player/src/mpv.rs)
All fields are logged at playback start so the log shows exactly what options were active. Key fields:
- `hwdec` — decoder selection (`auto`, `nvdec-copy`, `vulkan-copy`, etc.)
- `vf` — video filter string. Use `format=yuv420p` (or `auto`) for NVIDIA legacy stride fix.
- `video_sync` — `audio` (default), `display-resample` (locks to display refresh via `report_swap()`), `display-vdrop`, `display-adrop`, or `desync` (no A/V correction — debug option for isolating #39 passthrough dropout).
- `opengl_early_flush` — flush GL after each frame; may help with EGL pipeline ordering on NVIDIA.
- `video_latency_hacks` — compensates for imprecise Wayland vsync timestamps on NVIDIA 5xx legacy.
- `sub_scale`/`sub_pos`/`sub_respect_ass_styling`/`sub_color`/`sub_background` — subtitle appearance (Settings → Player). Scale/pos always applied (mpv's own defaults, harmless no-op if untouched); the other three only applied when non-default, since even a "default-looking" explicit value engages mpv's ASS-override machinery — see the Settings section of this doc and `Player::set_sub_style()`'s doc comment for the full mpv-manual-verified reasoning.

## Known Slint gotchas

These have each caused real bugs in this codebase:

**`Flickable` is the only reliable keyboard-scrollable container.** `ScrollView` ignores declarative `viewport-y` bindings (it manages its own scroll internally). `ListView` also writes to `viewport-y` from its own scroll handler, silently overwriting any binding you set. The correct pattern for any keyboard-driven scrollable list is `Flickable { viewport-height: ...; VerticalLayout { for ... } }` with `viewport-y` bound to a `clamp(...)` expression that tracks the focused index.

**Do not self-reference a `Flickable`'s own layout properties in its `viewport-y` binding.** Writing `viewport-y: clamp(... flk.height ... flk.viewport-height ...)` creates a binding whose dependencies Slint may not reliably track — `flk.height` and `flk.viewport-height` are layout-managed and may not trigger binding re-evaluation when `player-panel-cursor` changes. Instead, reference `parent.height` (the outer Rectangle's height) and the content layout's `preferred-height` directly: `clamp(-(cursor * 34px) + parent.height / 2 - 17px, min(0px, parent.height - list.preferred-height), 0px)`. This is what fixed the track panel scroll bug (#22).

**A `viewport-y` binding on a `Flickable` blocks native mouse-wheel scrolling.** When `viewport-y` is bound to an expression, the Flickable's internal scroll handler can't write to it (the binding overrides any assignment on the next frame), so mouse-wheel does nothing. When both keyboard nav and mouse-wheel scroll are needed: remove the `viewport-y` binding; on the outer Rectangle declare `property <length> kb-y: clamp(...)` and `changed kb-y => { fl.viewport-y = kb-y; }` for keyboard nav; the Flickable then handles mouse-wheel natively. Also fix any `fl.height` / `self.viewport-height` self-references in the old binding expression — use the outer container's `self.height` and the content layout's `preferred-height` instead. This is the pattern applied to all scrollable Flickables in the codebase (player panels, detail, series, home/movies/TV dashboards, library grid, settings right pane, browse list). Note: the browse list previously used `interactive: false` which blocks mouse-wheel regardless of bindings; changing it to `interactive: true` re-enables native scroll while child `TouchArea` clicks still fire normally (Slint distinguishes drag from click).

**Plain `Rectangle` children are horizontally centred by default.** If you need a fill bar or overlay anchored to the left edge, you must set `x: 0` explicitly. Omitting it centres the element and produces the "progress bar starts from the middle" bug.

**`KeyEvent.repeat` is unreliable — never use it to guard state transitions.** In practice `repeat` can be `false` for auto-repeated key events (confirmed on desktop Wayland, not just wireless keyboards). A guard like `if !event.repeat { close_screen() }` will fire on every spurious non-repeat event during a hold, chaining through screens unexpectedly. The correct pattern is to let the state machine be the guard: once the transition fires (e.g. `show-browse = false`), the outer `if AppState.show-browse` condition stops subsequent events from re-firing it. For search fields specifically: Backspace should only delete characters; use Escape as the dedicated "exit search" key. Never use `!event.repeat` to gate a backspace-exits-search path — a held Backspace will empty the query and then bleed into the close-screen handler.

**Slint ternary short-circuits dependency tracking.** If a property binding uses `condition ? A : B` and `B` contains a reactive property (e.g. `has-hover`), Slint only tracks `B`'s dependencies when the else-branch is actually evaluated. If the condition is initially true, `has-hover` is never read and hover changes never trigger a repaint. Fix: read the property unconditionally first using a block expression — `background: { let hov = ta.has-hover; cond ? Theme.accent : (hov ? Theme.surface : transparent) };`. This was the root cause of settings left-pane hover not working.

**`invoke_from_event_loop` closures must be `'static + Send`.** Capture owned values (`String`, `Arc<…>`) not references. This is the correct pattern for communicating from Tokio tasks back to Slint UI state.

**`TouchArea.moved` fires only during drag (button held), not plain cursor movement.** To react to mouse movement without a button press, use `changed mouse-x => { ... }` and `changed mouse-y => { ... }` callbacks. This is how the player controls overlay auto-show is implemented.

**`opacity: 0` elements remain fully hit-testable.** Setting `opacity: 0` makes an element invisible but it still participates in hit-testing and determines the mouse cursor shape — only `visible: false` removes it from event handling. The player controls bar fades via `opacity`, so its child `TouchArea`s were silently overriding `mouse-cursor: none` on the element beneath them. The fix is a full-size cursor-hider `TouchArea` declared last (highest z-order) with `enabled: !root.controls-visible` and `mouse-cursor: MouseCursor.none`. When `enabled: false`, a `TouchArea` passes events through to elements below it.

**`self.width` / `self.height` inside a conditional element reads from the parent layout cache — do not use it in ChangeTracker properties.** When a `changed prop => { ... }` ChangeTracker initialises it immediately reads the tracked property. If that property reads `self.width` or `self.height`, and `self` is an element inside a VerticalLayout/HorizontalLayout that is a conditional (`if cond: Element { ... }`), then `self.width`/`self.height` is derived from the parent layout's cache. If that layout cache is currently being evaluated — e.g. because a `kb-y` anchor position triggered it — and the layout cache called `ensure_updated()` on this very conditional (which then ran `user_init` → ChangeTracker → reads `self.width` → reads the layout cache) — Slint detects recursion and panics. **Fix:** replace `self.width`/`self.height` with `root.width`/`root.height` (the component root's size, set by the outer parent, never derived from the internal layout cache). Both values are always equal when the conditional element fills the full component width/height, which is the normal case. This was the root cause of the series screen "Recursion detected" crash — season tabs `kb-x` read `self.width`, which came from the content VerticalLayout's layout cache that was in the middle of calling `ensure_updated()` on the season tabs conditional.

**A sibling `Rectangle` with `width`/`height` bound to `root.width + Npx` / `root.height + Npx` (an *outset*, not flush) can also trigger "Recursion detected" during mouse-hover hit-testing.** `MediaCard`'s focus-ring overlay (see `widgets.slint`) was briefly changed from flush-with-root (`x:0; y:0; width:root.width; height:root.height;`) to a 3px outset (`x:-3px; y:-3px; width:root.width+6px; height:root.height+6px;`) so the ring wouldn't overlap the poster image/title text. This crashed live during hover — `properties.rs:583: Recursion detected`, inside `item_geometry` evaluation triggered by `send_mouse_event_to_item` — even though `root.width`/`root.height` is exactly the pattern the *previous* gotcha recommends as the safe fix, and this Rectangle isn't inside a conditional at all. Root cause not fully isolated (the backtrace only pointed at an internal, auto-numbered `Opacity_...` wrapper, not directly at this binding), but reverting the outset back to flush-with-root immediately and reproducibly fixed it. **Takeaway:** don't assume `root.width ± constant` is automatically as safe as bare `root.width` — if a card/row's focus ring needs to visually clear its own content, inset the *content* inward within the element's existing, already-stable bounds instead of expanding the ring outward past them; this also sidesteps a second, independent problem outsetting has (a `Flickable`-scrolled container clips content that intentionally overflows an item's own declared bounds, so an outset ring gets visibly cut off for edge-of-row items regardless of the crash). **Part 2 — the "inset the content" fix has its own trap:** doing that inset by giving the content `VerticalLayout` explicit geometry bindings (`x:3px; y:3px; width: root.width - 6px; height: root.height - 6px;`) crashed on the HTPC with the identical `properties.rs:583: Recursion detected` panic (`fjord.log` r716, triggered ~9s into playback when a WS delta refresh rebuilt the Continue Watching row's model). The dev machine never reproduced it — the difference is the *instantiation site*: `LibraryGrid` creates `MediaCard`s with `width:` only, no explicit `height:`, so `root.height` there is assigned by the enclosing row `HorizontalLayout`'s layout cache; an explicit-geometry binding on a child **layout element** that reads `root.height` feeds back into the same constraint computation that produces `root.height` → cycle. `SectionRow` passes `height:` explicitly (a plain property, no cache), which is why dashboards were immune and dev-machine testing passed. **Fix:** inset via `padding: 3px` on the layout (a native layout property, resolved entirely inside the layout's own cache — no external geometry binding), and have sized children read `root.height - constant` (plain-Rectangle children reading root has been stable for months) rather than `parent.height` (the padded layout, whose height participates in layout-info). General rule: never put explicit `x/y/width/height` bindings derived from `root` geometry on a **layout element** inside a reusable component — some instantiation site will eventually size the component through a layout cache and close the cycle.

**`parent` inside a reusable child component refers to that component's own immediate parent, not the grandparent two levels up.** `widgets.slint`'s `FadeInTrigger` (dropped into a screen as `fade := FadeInTrigger {}` to fade it in via `opacity: fade.fade-opacity`) originally wrote `parent.opacity = 1` directly inside its own `Timer.triggered`. Since the `Timer` is nested one level inside `FadeInTrigger`, and `FadeInTrigger` is nested one level inside the actual host screen, `parent` from the Timer's perspective resolved to `FadeInTrigger` itself (which has no meaningful `opacity` effect on anything), not the host screen two levels up — the write silently no-oped and every screen using it stayed stuck at `opacity: 0` forever (confirmed via a real screenshot showing a fully black content area, not a Slint compile error — `parent.opacity = 1` type-checks fine against `FadeInTrigger`'s own built-in `opacity` property). **Fix:** never rely on `parent` chaining out of a reusable child component to reach an ancestor two-plus levels up. Instead expose an `out property` on the child (`out property <float> fade-opacity: 0;`) and have the ancestor bind to it by name (`opacity: fade.fade-opacity;`) — this works regardless of nesting depth since it's a direct property reference, not a parent walk. **Also note:** `if`-gated element ids (`fade := FadeInTrigger {}` written identically inside N different `if cond: Element {...}` blocks) are NOT scoped per-conditional — Slint requires every named id to be unique across the whole enclosing component, so reusing the same id text in more than one `if` block is a compile error ("duplicated element id"), even though each block only ever instantiates one of them at a time. Every such usage needs a distinct id (e.g. `fade1`, `fade2`, ...).

**Returning a brand-new `ModelRc` from a "refresh this row" helper makes Slint destroy and recreate every delegate element, even when every row's data is identical.** A `Repeater`/`for` loop tracks the *model instance* it's bound to; swapping in a different `ModelRc` (e.g. `ModelRc::new(VecModel::from(rows))`) — rather than mutating the existing one — is indistinguishable from "everything changed" to Slint, so every child element (here, each card's poster `Image`) gets torn down and rebuilt from scratch. This silently defeated the whole point of `refresh_row_preserving_posters`/`upsert_cards_in_model` (Phases 91/93/94): they correctly carried the *poster data* forward into the new model, but the new model instance itself still forced every `Image` to be recreated, re-running `FadeInTrigger`'s fade-in (Phase 92) for cards whose poster never actually changed — a real flash, just one level down from where the fix was aimed. **Fix:** when the refreshed data has the same ids in the same order as what's already there (check first), call `existing_model.set_row_data(i, new_card)` for each index and return the *same* `ModelRc` — this fires a per-row change notification that Slint applies to the existing delegate in place, so unrelated conditional children (like the poster `Image`) are never destroyed. Only fall back to building a new model when the row *membership or order* genuinely changed (a real content difference, where recreating elements is correct and expected, not a bug). **Phase 96 note:** this same same-shape/mutate-in-place bug turned out to be duplicated across five different "build a `Vec<CardItem>`, push it to a model" call sites (`poster.rs`'s two poster-decode functions, `movies.rs`'s library-decode function, plus the two already-fixed here) — fixing it once wasn't enough because the underlying mistake (a fresh `ModelRc::new(VecModel::from(...))` with no same-shape check) had been independently reinvented in each file. Now centralized in `main.rs::apply_cards_preserving_identity`, next to `item_to_card_item`/`items_to_model` — any new "refresh this row" code should call it rather than writing the check again. **Phase 97 note:** even after fixing all five, the library grid *still* flashed — `browse::refresh_library_display` builds `library-display` (the model the grid actually renders) as its own independent derived view over `all_movies`/etc, via sort+filter+search, and had never been touched — so `all_movies` itself was correctly preserved while the thing on screen was rebuilt from scratch anyway. **Lesson generalized:** when a value flows through more than one derived/rendered model (a backing list *and* a separately-sorted/filtered display list), every hop needs the same-shape check independently; fixing the source model doesn't help if a later stage still does a blind rebuild. **Phase 98 note:** `apply_cards_preserving_identity`'s same-shape check only stops Slint from destroying/recreating card *elements* — it says nothing about whether the `Image` *value* newly bound to `source:` is the same object as before. `movies.rs::push_library_cards` and `poster.rs`'s two poster-decode functions always built a fresh `slint::Image::from_rgba8(...)` from newly-decoded bytes regardless of whether the row already had one, so even a `same_shape=true` apply still swapped every visible card's texture in one synchronous batch — visibly disruptive on a large grid with zero element recreation involved. Fix: look up the existing `CardItem` by id first and reuse its `poster` handle directly whenever `has_poster` is already true; only decode a new `Image` for a row that didn't have one. Applies whenever a function *produces* a decoded image (not just merges one that's already decoded, which `refresh_row_preserving_posters`/`refresh_favorites` already did correctly from the start). **Phase 99 note:** even with identity and Image-handle reuse both fixed, Movies/Collections/Music still flashed while TV never did — traced (via `#[track_caller]` added to `refresh_library_display` and `update_library_filter`, logging which of ~15 call sites fired) to `movies.rs::push_library_cards` setting `library-display` **directly** to its just-applied model — which is in raw network/decode order — instead of routing through `browse::refresh_library_display`'s sort/filter/query pipeline like `poster.rs::push_decoded_series` (TV's equivalent) always correctly did. The grid briefly showed Jellyfin's own raw order, indistinguishable from sorted for most items, then visibly reshuffled the next time anything called `refresh_library_display` for real (a nav double-click's `library-search-clear`, a sort-bar apply) — because Jellyfin's ordering and Rust's `.to_lowercase()` title comparator disagree on at least one tied pair (articles/symbols/accents), and stable-sort tie-breaking over a different starting order shifts more than just that one item. **Lesson generalized further:** identity-preservation (Phase 96) and value-reuse (Phase 98) both assume the *content* being applied is already correct — neither one catches a shortcut that skips the actual sort/filter computation and substitutes a merely-similar-looking order instead. When two code paths are supposed to produce "the same" derived view, one of them taking a shortcut that happens to usually agree with the other is worse than both doing the full computation, because the disagreement only surfaces as an intermittent, hard-to-repro flash rather than a compile error or an always-wrong result. **Phase 100 note:** immediately after Phase 99, a new report — Collections' unplayed-count badge flashed in then vanished. `main.rs::item_to_card_item` correctly copies `unplayed_count` from `MediaItem.user_data.unplayed_item_count` (Jellyfin populates this for any folder-like item, BoxSets included, not just Series), so `refresh_row_preserving_posters`' metadata-merge pass showed the badge correctly. But `movies.rs`'s `meta`/`decoded` tuples (the *other* landing point, firing moments later once posters finish decoding) never carried an unplayed-count field at all — `CardItem::default()` leaves it 0 — so that pass's same-shape mutate-in-place `set_row_data` overwrote the just-correct value back to 0, even though nothing about row identity had changed. `poster.rs::push_decoded_series` (TV) never had this gap since its tuple already threaded the field through. **Same lesson as Phase 99, one field lower:** it's not enough for two code paths to agree on *whether* to rebuild vs. mutate-in-place (Phase 96) or *which* Image/order to use (Phases 98-99) — every field on the shared struct has to actually be threaded through both paths, or the one that drops a field will periodically stomp the one that doesn't, and it'll look like a display bug when it's actually a data-plumbing gap.

**`vertical-alignment: bottom` combined with `overflow: elide` + `wrap: word-wrap` mis-measures text height in Slint 1.16.1, as soon as the text actually wraps to 2+ lines — regardless of whether truncation is needed.** `MediaCard`'s title `Text` (widgets.slint) originally used a fixed 34px (2-line) box with default (top) alignment; a short, 1-line title left blank space at the bottom of that box before the subtitle/year started. Bottom-aligning the title (`vertical-alignment: bottom`) fixed that specific case. But once a *long* title wrapped to 2 real lines — with or without a 3rd line being elided away — the same gap reappeared, and only for some titles, which made it look content-dependent rather than systemic. Root-caused with a disposable standalone Slint crate outside the repo (pinned to the exact `slint = "=1.16.1"` from this project's `Cargo.lock`, since the bug does *not* reproduce on 1.17.1 — the two versions clearly handle this differently) reproducing the two real offending titles pulled straight from `~/.cache/fjord/*.json` (`Chillin' in Another World With Level 2 Super Cheat Powers`, `Your Friendly Neighborhood Spider-Man`) at the real card width, with `spectacle -b -n -a -o <path>` (KDE's screenshot CLI) to inspect the rendered output directly — confirmed that `vertical-alignment: top` closes the gap correctly (its baseline is hardcoded to `Font::Length::zero()` in `i-slint-core`'s `textlayout.rs`, no text-height computation involved at all) while `bottom` does not, even though both are given an identical 2-line-filling box. **Fix (and a design change in the same pass, per user preference discovered along the way — don't cap the title at all, always show it in full):** dropped the fixed height / `overflow: elide` / bottom-alignment approach entirely. Both the title and subtitle `Text` elements are now plain `wrap: word-wrap` with no explicit height, so they grow to however many lines they actually need (no more silent truncation of long titles or long episode names) and default to top alignment, which is immune to this bug by construction. `poster-rect` reads the sibling text block's `preferred-height` (named `text-col`) and reserves exactly that much space (`max(0px, root.height - 12px - text-col.preferred-height)`), so the poster shrinks — down to fully hidden in a pathological case — rather than the text ever overflowing the card's own bounds. This still reads `root.height` only from a plain `Rectangle` (the established-safe pattern) and reads a *sibling's* `preferred-height`, not an ancestor's layout cache, so it doesn't reintroduce the `LibraryGrid` "Recursion detected" crash from the gotcha above — re-verified with the same disposable-crate technique, instantiating the equivalent structure both with an explicit height (`SectionRow`-style) and inside a `HorizontalLayout` row with no explicit height (`LibraryGrid`-style, the historically vulnerable case) plus a hover-triggered focus ring (the exact mechanism of the earlier real crash): no panic either way. **General lesson:** when a `vertical-alignment` fix works for the simple case (single line) but a report comes back claiming the *same symptom* on a different, more complex input, don't assume it's the same bug with a missed edge case — reproduce the exact input in isolation before re-patching, since here the "same-looking" gap on wrapped titles had a completely different (and version-specific) root cause from the one-line case, and no amount of staring at the single-line fix would have surfaced it.

**A stray trailing newline/whitespace in Jellyfin's scraped `Overview` metadata inflates `StorylineSection`'s expanded-box height beyond what the visible text needs — this is correct Slint behavior on unclean input data, not a rendering bug.** User report: one specific series' expanded overview left a visible gap between the last line of text and the season tabs below it; every other item's overview expanded correctly with no gap, which pointed at that item's data rather than the shared `StorylineSection` component (confirmed by a standalone repro of the component in isolation with the exact on-screen synopsis text — no gap, box tightly wraps the text). `wrap: word-wrap` `Text` respects embedded `\n` as a real line break, so a trailing `\n` or `\n\n` left over from whatever scraped `Overview` from TMDb/TVDB (or a similar provider) creates one or more genuinely blank lines at the end of the string — invisible (nothing to render), but still counted in `txt.preferred-height`, which is exactly what `StorylineSection`'s expanded height binds to (`max(txt.preferred-height, 66px)`). No code path anywhere trims this field before it reaches `AppState` — `.overview.clone().unwrap_or_default()` was used verbatim at every call site. **Fix:** added `.trim()` at all 8 sites across `detail.rs`, `series.rs` (×2), `season.rs`, `artist.rs`, `album.rs`, `collection.rs`, `person.rs` (bio) — the general fix (sanitize the data once, at the boundary where it enters the app) rather than a per-item or per-screen special case, consistent with how every other `StorylineSection` consumer shares the identical component and should share the identical data-cleanliness guarantee.

## Style

- Standard Rust formatting (`cargo fmt`)
- Errors: use `anyhow::Result` at the top level, `thiserror` for library error types
- No `unwrap()` in library code — propagate errors
- Keep `fjord-api` and `fjord-player` free of Slint imports
- Every `.rs` and `.slint` source file opens with a `// ── <crate> · <filename> ──` header block listing its major symbols/sections (one line each). Longer files additionally carry `// ──` inline section markers immediately before major functions and visual blocks. The header is the first thing in the file, before any `use` statements or declarations. Update the header whenever symbols are added, removed, or their behaviour changes — not just when the name changes.
